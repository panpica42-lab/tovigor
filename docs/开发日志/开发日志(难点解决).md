## 力量滑块拖动"不跟手"问题

**日期**：2026-02-10  
**涉及文件**：`pages/partTraining/formal-training.vue`

---

### 问题描述

在正式训练页的底部力量控制面板中，我们实现了一个自定义的力量滑块（5-55kg 范围），用于调节配重。

**初始状态的问题**：用户拖动滑块时，手指在前面走，滑块的 UI（填充条和圆形按钮）在后面追，有明显的延迟感，"不跟手"。

这在触摸交互中是非常糟糕的体验。

---

### 问题分析

#### 1. touchmove 事件的触发频率

- 普通 60Hz 设备：每 16.67ms 触发一次
- 高刷 120Hz 设备：每 8.33ms 触发一次
- 也就是说，手指滑动 1 秒会触发 **60-120 次** touchmove 事件

#### 2. 原始代码每次 touchmove 做了什么？

```javascript
const onSliderTouchMove = (e) => {
    if (sliderRect) {
        updateSliderValue(e.touches[0].clientX)
    }
}

const updateSliderValue = (touchX) => {
    // ...计算...
    currentWeight.value = newValue  // 触发 Vue 响应式！
}
```

每次 touchmove（每 8-16ms）都会触发完整的 Vue 响应式链条：

1. 计算新值（很快）
2. 修改 `ref` -> 触发依赖收集
3. `computed` 重新计算 `sliderFillWidth` 和 `sliderThumbLeft`
4. Vue 调度虚拟 DOM diff
5. patch 真实 DOM

**链路太重了！**

#### 3. CSS transition 的"拖尾"效果

```scss
.slider-thumb {
    transition: left 0.05s ease-out;  // 故意的 50ms 延迟！
}
```

这个 transition 本来是为了让 **按钮点击调节** 时有平滑动画，但在拖动时变成了累赘——UI 会故意落后手指 50ms。

---

### 第一次优化：移除拖动时的 transition

**思路**：transition 只在按钮点击时需要，拖动时应该禁用。

**实现**：

1. 添加 `isDragging` 状态标记拖动中
2. 拖动时通过 CSS class 禁用 transition

```javascript
const isDragging = ref(false)

const onSliderTouchStart = (e) => {
    isDragging.value = true
    // ...
}

const onSliderTouchEnd = () => {
    isDragging.value = false
}
```

```scss
.custom-slider.is-dragging .slider-fill,
.custom-slider.is-dragging .slider-thumb {
    transition: none !important;
}
```

**效果**：有一定改善，但链路过重的问题仍然存在。

---

### 第二次优化：节流 + CSS 变量（过渡方案）

> ⚠️ **注意**：这是历史过渡方案，后来被 **renderjs 终极方案** 替代。记录在此仅供参考，了解优化思路的演进过程。

**核心思路**：

1. **降低处理频率**：节流到 30Hz（33ms 一次），中间的事件直接丢弃
2. **拖动时绕过响应式**：用普通变量存值，通过 CSS 变量直接控制视觉
3. **松手时才同步数据**：只在 touchend 时触发一次 Vue 响应式更新

#### 实现细节

**1. 节流控制**

```javascript
let lastUpdateTime = 0
const THROTTLE_INTERVAL = 33 // 30Hz

const onSliderTouchMove = (e) => {
    // 节流：33ms 内只处理一次
    const now = Date.now()
    if (now - lastUpdateTime < THROTTLE_INTERVAL) return
    lastUpdateTime = now
    
    updateSliderVisual(e.touches[0].clientX)
}
```

**2. 绕过响应式，直接操作 CSS 变量**

```javascript
// 普通变量，不触发响应式
let dragValue = 30

const updateSliderVisual = (touchX) => {
    // 计算百分比
    const percent = offsetX / sliderRect.width
    dragValue = Math.round(sliderMin + percent * (sliderMax - sliderMin))
    
    // 直接设置 CSS 变量，不走 Vue
    const sliderEl = sliderContainerRef.value?.$el || sliderContainerRef.value
    if (sliderEl) {
        sliderEl.style.setProperty('--drag-percent', `${percentValue}%`)
        sliderEl.style.setProperty('--drag-transform', `translateX(calc(${percentValue}cqw - 24rpx))`)
    }
}
```

**3. 样式中使用 CSS 变量覆盖 Vue 绑定**

```scss
.custom-slider {
    --drag-percent: 0%;
    --drag-transform: translateX(0);
}

/* 拖动时使用 CSS 变量，覆盖 Vue 的 :style 绑定 */
.custom-slider.is-dragging .slider-fill {
    width: var(--drag-percent) !important;
    transition: none !important;
}

.custom-slider.is-dragging .slider-thumb {
    transform: var(--drag-transform) !important;
    transition: none !important;
}
```

**4. 松手时才同步到响应式数据**

```javascript
const onSliderTouchEnd = () => {
    if (isDragging.value) {
        // 只在这里触发一次 Vue 响应式更新
        currentWeight.value = dragValue
    }
    sliderRect = null
    isDragging.value = false
}
```

---

### 优化效果对比（过渡方案 vs 优化前）

| 阶段 | 操作 | 触发响应式？ | 频率 |
|------|------|-------------|------|
| **优化前** touchmove | 计算 -> 改 ref -> Vue diff -> DOM | 是 | 60-120Hz |
| **过渡方案** touchmove | 计算 -> 改 CSS 变量 | 否 | 30Hz |
| **过渡方案** touchend | 同步 `currentWeight.value` | 是 | 1次 |

> 这个方案有改善，但仍有 **~16-32ms 的跨线程延迟**，因为 touchmove 事件仍需从渲染层传到逻辑层。最终被 renderjs 方案替代（见下文）。

---

### 经验总结（通用）

1. **transition 是双刃剑**：对按钮点击友好，对拖动是累赘，需要区分场景
2. **响应式系统有代价**：高频更新场景下，要考虑绕过响应式直接操作 DOM
3. **不是所有事件都需要处理**：30Hz 的视觉刷新率对于滑块来说完全够用
4. **用 `transform` 替代 `left`**：前者只触发 composite，后者会触发 layout
5. **CSS 变量是好帮手**：可以在 JS 和 CSS 之间搭建轻量级的通信桥梁

## Passive Event Listener 问题

**日期**：2026-02-10  
**涉及文件**：`pages/partTraining/formal-training.vue`

---

### 问题描述

在 H5 环境下测试滑块时，控制台报出警告：

```
Ignored attempt to cancel a touchmove event with cancelable=false, 
for example because scrolling is in progress and cannot be interrupted.

Unable to preventDefault inside passive event listener due to target 
being treated as passive.
```

尽管我们在模板中使用了 `@touchmove.stop.prevent`，`preventDefault()` 仍然不生效。

---

### 问题分析

现代浏览器（Chrome 56+）为了优化滚动性能，默认将 `touchstart` 和 `touchmove` 事件标记为 **passive**。

在 passive 模式下：
- 浏览器不会等待 JS 执行完毕再决定是否滚动
- 可以立即开始滚动，提升流畅度
- 但代价是 `preventDefault()` 被禁用

这意味着 Vue 的 `.prevent` 修饰符失效了。

---

### 解决方案

使用 CSS `touch-action` 属性告诉浏览器这个元素不需要默认的触摸行为：

```scss
.custom-slider {
    touch-action: none; /* 阻止默认触摸行为 */
}
```

`touch-action: none` 的含义：
- 禁用所有默认触摸操作（滚动、缩放等）
- 浏览器不再需要等待判断，直接跳过默认行为
- 完全由 JS 接管触摸事件

---

### 其他可选值

| 值 | 含义 |
|---|------|
| `auto` | 默认，浏览器决定 |
| `none` | 禁用所有默认触摸行为 |
| `pan-x` | 只允许水平滚动 |
| `pan-y` | 只允许垂直滚动 |
| `manipulation` | 允许滚动和缩放，但禁用双击缩放 |

---

### 为什么 Android App 也有这个问题？

你可能会疑惑：我们的项目是运行在 Android 设备上的 App，跟浏览器有什么关系？

**关键点**：uni-app 的普通 `.vue` 页面在 Android App 上运行时，**并不是原生渲染**，而是通过 **WebView**（Android 内置的浏览器组件）来渲染的。

```
┌─────────────────────────────────────┐
│           Android App               │
│  ┌───────────────────────────────┐  │
│  │          WebView              │  │  ← 内嵌浏览器引擎（Chromium 内核）
│  │  ┌─────────────────────────┐  │  │
│  │  │   你的 Vue 页面代码      │  │  │  ← 在 WebView 里运行
│  │  └─────────────────────────┘  │  │
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘
```

WebView 使用的是 **Chromium 内核**，和 Chrome 浏览器一样，所以 Chrome 浏览器的 passive event listener 机制，WebView 里也有。

| 方案 | 渲染方式 | 有这个警告吗？ |
|------|----------|----------------|
| 普通 `.vue` 页面 | WebView（H5渲染） | ✅ 有 |
| `.nvue` 页面 | 原生渲染（Weex） | ❌ 没有 |
| 纯原生 Kotlin/Java | 原生渲染 | ❌ 没有 |

**总结**：uni-app 的 App ≠ 纯原生 App，它是「原生壳 + WebView 渲染页面」的混合架构，所以浏览器的特性（包括 passive event listener）都会生效。

---

### 经验总结

1. **`.prevent` 修饰符不是万能的**：passive event listener 会让它失效
2. **优先用 CSS 解决**：`touch-action` 比 JS 的 `preventDefault()` 更可靠
3. **只在需要的元素上使用**：不要全局禁用触摸行为，会影响正常滚动

## 力量滑块性能优化 - renderjs 终极方案

**日期**：2026-02-11  
**涉及文件**：`components/ui-box/power-slider.vue`、`pages/partTraining/formal-training.vue`

---

### 背景

在之前的优化中（节流 30Hz + CSS 变量），滑块跟手性已有改善，但在真机测试中仍能感受到轻微延迟。

这是因为 **uni-app 的双线程架构** 导致的固有延迟：

```
┌─────────────────┐     ┌─────────────────┐
│   渲染层        │ ←→  │   逻辑层        │
│   (WebView)     │     │   (V8 引擎)     │
│   DOM/CSS       │     │   Vue 代码      │
└─────────────────┘     └─────────────────┘
        ↑                       ↑
        │    ~16-32ms 通信延迟   │
        └───────────────────────┘
```

每次 touchmove 事件从渲染层传到逻辑层，处理后再传回渲染层更新 DOM，至少有 **16-32ms 的往返延迟**。

---

### 解决方案：renderjs

renderjs 是 uni-app 提供的一种机制，允许在 **渲染层直接执行 JS 代码**，从而绑过跨线程通信。

#### 核心思路

1. 将触摸事件处理完全放在 renderjs（渲染层）
2. 在渲染层直接操作 DOM 更新视觉
3. 只在松手或值变化时才回调逻辑层同步数据

---

### 踩坑记录

#### 坑 1：模板直接绑定 renderjs 方法无效

```vue
<!-- 错误写法 -->
<view @touchstart="bindSlider.onTouchStart">

<!-- 这样绑定在 App 环境下不生效 -->
```

**正确做法**：在 renderjs 中通过 `addEventListener` 手动绑定原生事件

#### 坑 2：renderjs 中 `this.$el` 不存在

```javascript
// 错误
mounted() {
    this.bindEvents(this.$el)  // this.$el 是 undefined
}
```

**正确做法**：通过 `onDataChange` 回调的 `instance` 参数获取 DOM 元素

```javascript
// 正确
onDataChange(newData, oldData, ownerInstance, instance) {
    if (!this.bindDone && instance && instance.$el) {
        this.bindDone = true
        this.bindEvents(instance.$el)
    }
}
```

#### 坑 3：视觉被整数值"卡住"导致跳格

```javascript
// 错误：视觉更新被整数值约束
const newValue = Math.round(...)
if (newValue !== this.currentValue) {
    // 只有值变化才更新 DOM → 跳格
    this.fillEl.style.transform = `scaleX(${scale})`
}
```

**正确做法**：视觉每帧更新（连续），业务值离散回调

```javascript
// 正确：视觉与业务分离
updatePosition(touchX) {
    const scale = offsetX / this.sliderRect.width
    
    // 视觉每次都更新（连续）
    this.fillEl.style.transform = `scaleX(${scale})`
    this.thumbEl.style.transform = `translateX(${thumbOffset}px)`
    
    // 业务值只有变化时才回调（离散）
    const newValue = Math.round(this.min + scale * (this.max - this.min))
    if (newValue !== this.currentValue) {
        this.currentValue = newValue
        this.ownerInstance.callMethod('onDragUpdate', newValue)
    }
}
```

#### 坑 4：addEventListener 的 this 指向问题

```javascript
// 错误：直接传方法引用
el.addEventListener('touchstart', this.onTouchStart, { passive: false })
```

在原生 JS 中，事件回调里的 `this` 默认指向触发事件的 DOM 元素，而不是 renderjs 模块对象。这会导致 `this.disabled`、`this.isDragging` 等属性访问错误。

虽然某些 uni-app 版本可能自动 bind，但不同版本/平台行为不一致，不能依赖。

**正确做法**：用箭头函数捕获 this，并保存引用方便卸载

```javascript
// 正确：箭头函数 + 保存引用
bindEvents(el) {
    // 箭头函数捕获外层 this
    this._onStart = (e) => this.onTouchStart(e)
    this._onMove = (e) => this.onTouchMove(e)
    this._onEnd = (e) => this.onTouchEnd(e)
    
    el.addEventListener('touchstart', this._onStart, { passive: false })
    el.addEventListener('touchmove', this._onMove, { passive: false })
    el.addEventListener('touchend', this._onEnd, { passive: false })
}

// 卸载时可以正确移除
unbindEvents(el) {
    el.removeEventListener('touchstart', this._onStart)
    el.removeEventListener('touchmove', this._onMove)
    el.removeEventListener('touchend', this._onEnd)
}
```

**为什么箭头函数有效**：箭头函数没有自己的 `this`，它会捕获定义时的外层作用域的 `this`。在 `bindEvents` 方法中，外层 `this` 就是 renderjs 模块对象。

#### 坑 5：rpx 单位在不同设备上的像素值不同

```javascript
// 错误：写死像素值
this.thumbEl.style.transform = `translateX(${thumbOffset - 12}px)`
// 48rpx 的一半在 375px 设备上是 24px，但在 414px 设备上约 26.5px
```

**正确做法**：在渲染层实际测量元素尺寸

```javascript
bindEvents(el) {
    this.thumbEl = el.querySelector('.slider-thumb')
    
    // 测量 thumb 实际像素宽度
    if (this.thumbEl) {
        const thumbRect = this.thumbEl.getBoundingClientRect()
        this.halfThumbPx = thumbRect.width / 2
    }
}

updatePosition(touchX) {
    const thumbOffset = scale * this.sliderRect.width
    // 使用动态测量值
    this.thumbEl.style.transform = `translateX(${thumbOffset - this.halfThumbPx}px)`
}
```

#### 坑 6：松手瞬间滑块颤动 - 双重控制冲突

**日期**：2026-03-20

**问题现象**：拖动滑块松手时，thumb 会有轻微的位置跳动/颤动。

**问题分析**：

```
拖拽中 ──────────────────────────────────────────────► 松手后
   │                                                      │
   ▼                                                      ▼
renderjs 直接设置 DOM                              逻辑层 thumbStyle 生效
style.transform = translateX(156.78px)     →      transform: translateX(calc(50cqw - 24rpx))
   │                                                      │
   └──────── 精确像素值 ────────┘     └──────── cqw/rpx 单位转换 ──────┘
                                              ↓
                                    微小精度差异 = 颤动！
```

松手时的完整链路：
1. `onTouchEnd` 调用 `callMethod('onDragEnd', value)`
2. 逻辑层 `emit('update:modelValue')` 更新响应式数据
3. Vue 触发 `thumbStyle` computed 重新计算（用 `cqw` 单位）
4. 模板的 `:style="thumbStyle"` 覆盖了 renderjs 设置的精确 `px` 值
5. 两种计算方式的微小精度差异导致位置跳动

**为什么逻辑层用 `cqw` 而不是 `px`**：逻辑层无法访问 DOM，拿不到 `getBoundingClientRect()`，只能用相对单位让浏览器计算。这是 uni-app 双层架构的固有限制，不是设计失误。

**解决方案**：App 平台让 renderjs **完全接管** fill 和 thumb 的位置控制，逻辑层的 `:style` 只在 H5 平台生效。

```vue
<!-- App 平台：不绑定 :style，完全由 renderjs 控制 -->
<!-- #ifdef APP-PLUS -->
<view class="slider-fill"></view>
<view class="slider-thumb"></view>
<!-- #endif -->

<!-- H5 平台：使用逻辑层的 :style（fallback） -->
<!-- #ifndef APP-PLUS -->
<view class="slider-fill" :style="fillStyle"></view>
<view class="slider-thumb" :style="thumbStyle"></view>
<!-- #endif -->
```

同时在 renderjs 的 `onDataChange` 中，非拖拽状态下也由 renderjs 同步位置：

```javascript
onDataChange(newData, oldData, ownerInstance, instance) {
    // ... 数据同步 ...
    
    // 非拖拽状态下，由 renderjs 统一控制位置（处理外部 modelValue 变化）
    if (!this.isDragging && this.fillEl && this.thumbEl) {
        this.syncPositionFromValue()
    }
}
```

**经验总结**：当 renderjs 接管 DOM 控制时，要**彻底接管**，不能让逻辑层的响应式绑定再插手，否则两套计算方式的精度差异会导致视觉问题。

---

### 最终架构

```
┌─────────────────────────────────────────────────────────┐
│                    渲染层 (WebView)                      │
│                                                          │
│   touchstart/move ──→ updatePosition()                  │
│                           │                              │
│                           ├─→ 计算 scale (0~1)           │
│                           │                              │
│                           ├─→ fill.style.transform  ← 每帧更新(连续) │
│                           ├─→ thumb.style.transform      │
│                           │                              │
│                           └─→ if (值变化) callMethod() ← 离散回调 │
│                                    ↓                     │
└────────────────────────────────────┼────────────────────┘
                                     ↓ (仅值变化时)
┌────────────────────────────────────┼────────────────────┐
│                    逻辑层 (V8)                           │
│                                    ↓                     │
│                         onDragUpdate(value)             │
│                               ↓                          │
│                    emit('dragging', value) → 显示数字    │
└─────────────────────────────────────────────────────────┘
```

### 关键代码

#### renderjs 部分

```javascript
// components/ui-box/power-slider.vue
<script module="bindSlider" lang="renderjs">
export default {
    data() {
        return {
            sliderEl: null,
            fillEl: null,
            thumbEl: null,
            sliderRect: null,
            currentValue: 30,
            min: 5,
            max: 55,
            disabled: false,
            isDragging: false,
            ownerInstance: null,
            bindDone: false
        }
    },
    methods: {
        // 通过 prop 变化获取 DOM 元素
        onDataChange(newData, oldData, ownerInstance, instance) {
            this.ownerInstance = ownerInstance
            if (newData) {
                this.currentValue = newData.value
                this.min = newData.min
                this.max = newData.max
                this.disabled = newData.disabled
            }
            
            // 只绑定一次事件
            if (!this.bindDone && instance && instance.$el) {
                this.bindDone = true
                this.bindEvents(instance.$el)
            }
        },
        
        bindEvents(el) {
            this.sliderEl = el
            this.fillEl = el.querySelector('.slider-fill')
            this.thumbEl = el.querySelector('.slider-thumb')
            
            el.addEventListener('touchstart', this.onTouchStart, { passive: false })
            el.addEventListener('touchmove', this.onTouchMove, { passive: false })
            el.addEventListener('touchend', this.onTouchEnd, { passive: false })
        },
        
        onTouchStart(e) {
            e.preventDefault()
            if (this.disabled) return
            
            this.isDragging = true
            this.sliderRect = this.sliderEl.getBoundingClientRect()
            this.updatePosition(e.touches[0].clientX)
        },
        
        onTouchMove(e) {
            e.preventDefault()
            if (!this.isDragging || !this.sliderRect) return
            this.updatePosition(e.touches[0].clientX)
        },
        
        onTouchEnd(e) {
            e.preventDefault()
            if (!this.isDragging) return
            
            this.isDragging = false
            this.ownerInstance.callMethod('onDragEnd', this.currentValue)
        },
        
        updatePosition(touchX) {
            let offsetX = touchX - this.sliderRect.left
            offsetX = Math.max(0, Math.min(offsetX, this.sliderRect.width))
            
            const scale = offsetX / this.sliderRect.width
            
            // 视觉每帧更新
            this.fillEl.style.transform = `scaleX(${scale})`
            this.thumbEl.style.transform = `translateX(${scale * this.sliderRect.width - 12}px)`
            
            // 业务值离散回调
            const newValue = Math.round(this.min + scale * (this.max - this.min))
            if (newValue !== this.currentValue) {
                this.currentValue = newValue
                this.ownerInstance.callMethod('onDragUpdate', newValue)
            }
        }
    }
}
</script>
```

---

### 性能对比

| 指标 | 优化前（逻辑层处理） | renderjs 方案 |
|------|---------------------|---------------|
| 触摸响应延迟 | ~16-32ms（跨线程） | **0ms**（渲染层直接） |
| 视觉更新 | 跳格（整数约束） | **连续**（scale 驱动） |
| CSS 开销 | width 变化（layout+paint） | **transform**（composite only） |
| 跨线程通信 | 每帧 | **仅值变化时** |

### 平台兼容

renderjs 只在 **App 平台** 生效，H5 平台需要 fallback：

```vue
<!-- #ifdef APP-PLUS -->
<script module="bindSlider" lang="renderjs">
// App 平台：renderjs 处理
</script>
<!-- #endif -->

<!-- #ifndef APP-PLUS -->
<script module="bindSlider" lang="renderjs">
// H5 平台：占位模块，逻辑层处理
export default {
    methods: {
        onDataChange() {}
    }
}
</script>
<!-- #endif -->
```

### 经验总结

1. **了解框架的线程模型**：uni-app 双线程架构的跨线程延迟是固有的
2. **renderjs 是性能利器**：对于高频交互，把逻辑放在渲染层能消除延迟
3. **正确获取 DOM**：renderjs 中用 `instance.$el`，不是 `this.$el`
4. **视觉与业务分离**：视觉连续更新，业务离散回调，两者解耦
5. **transform 优于 width/left**：只触发 composite，跳过 layout 和 paint
6. **平台差异要兼容**：renderjs 只在 App 生效，H5 需要 fallback

## PowerDial 旋钮组件优化（7 项）

**日期**：2026-03-23  
**涉及文件**：`components/ui-box/power-dial.vue`

---

### 背景

PowerDial 是自由训练页的核心控件，基于 renderjs + conic-gradient 实现。初版功能正确但存在多项性能和交互隐患，尤其在低端 Android 设备上可能出现掉帧和操作不精准。本次集中进行了 7 项优化。

---

### 优化 1：圆心计算偏差修复

**问题**：`onTouchStart` 中用根节点 `.power-dial` 的 `getBoundingClientRect()` 计算圆心，但弧形元素 `.dial-arc-progress` 有 `top: 20rpx` 偏移，导致视觉圆心和计算圆心有约 20rpx（5-10px）的 Y 轴偏差。越靠近 12 点钟方向，角度映射越"飘"。

**修复**：改用 `arcProgressEl`（弧形元素）取圆心，带 fallback 到 `dialEl`。

```javascript
// 修复前
const rect = this.dialEl.getBoundingClientRect()

// 修复后
const refEl = this.arcProgressEl || this.dialEl
const rect = refEl.getBoundingClientRect()
```

**原理**：`arcProgressEl` 的 `conic-gradient` 圆心（`at 50% 50%`）就是用户看到的视觉圆心，以它为基准计算角度更精准。以后调 `top` 值，交互自动跟着对。

---

### 优化 2：conic-gradient 改为 CSS 变量驱动

**问题**：每次 touchmove 都在 JS 中拼接整段 `conic-gradient(...)` 长字符串，然后赋给 `style.background`，导致浏览器每次都要重新解析整个背景声明。

**修复**：CSS 中用 `var(--progress-angle)` 占位，JS 中只改 CSS 变量。

```css
/* CSS 中写死结构 */
.dial-arc-progress {
    --progress-angle: 0deg;
    background: conic-gradient(
        from -120deg at 50% 50%,
        #00C853 0deg,
        #00C853 var(--progress-angle),
        transparent var(--progress-angle),
        transparent 240deg
    );
}
```

```javascript
// JS 中一行搞定
this.arcProgressEl.style.setProperty('--progress-angle', angle + 'deg')
```

**好处**：不拼字符串、不做整段背景重建、后期改颜色/结构更方便。

---

### 优化 3：filter: drop-shadow 移入 CSS 静态声明

**问题**：`updateDOM()` 每次 touchmove 都重复设置 `style.filter = 'drop-shadow(...)'`，但值从头到尾没变过，属于"无意义高频写样式"。`style.opacity` 也一样。

**修复**：

| 属性 | 改动 |
|------|------|
| `filter: drop-shadow(...)` | 移入 CSS 静态声明，不再在 JS 中操作 |
| `opacity` | 只在 `syncView()`（开关切换时）设置，`updateDOM()` 中移除 |

```css
.dial-arc-progress {
    opacity: 0;  /* 关机时隐藏 */
    filter: drop-shadow(0 0 8px rgba(0, 200, 83, 0.5));  /* 写死 */
}
```

`opacity: 0` 时 `drop-shadow` 不可见也不触发渲染，零额外开销。

**职责划分**：

| 位置 | 管什么 |
|------|--------|
| CSS | filter 写死，opacity 初始值 |
| syncView | 开关切换时管 opacity（0/1）+ 角度 |
| updateDOM | 拖动时**只更新角度**，一行搞定 |

---

### 优化 4：缓存 `.kg-value` DOM 元素引用（懒缓存策略）

**问题**：`updateDOM()` 每次 touchmove 都执行 `this.dialEl.querySelector('.kg-value')`，是高频 DOM 查询。

**难点**：`.kg-value` 元素在 `v-if` 控制下，关机时不存在，开机后 Vue 重建新 DOM，旧引用失效。

**修复**：三段式懒缓存策略。

```
1. bindEvents 时：尝试缓存（关机状态下拿到 null，没关系）
2. onDataChange 时：检测开关状态变化 → 清缓存
   if (this.powerOn !== newData.powerOn) { this.valueEl = null }
3. updateDOM 时：如果 valueEl 为 null，懒查询一次并缓存
   if (!this.valueEl) {
       this.valueEl = this.dialEl.querySelector('.kg-value')
   }
```

**效果**：每次开关周期只查询一次，后续拖动全用缓存。

---

### 优化 5：requestAnimationFrame 合帧

**问题**：每个 touchmove 事件都立即执行 `updateFromTouch` → `updateDOM`。手势事件触发频率可能高于屏幕实际刷新频率（60Hz = 16.67ms，但 touchmove 可能 8ms 触发一次），导致每帧做了多次无意义的 DOM 写入。

**修复**：touchmove 只存最新坐标 + 请求 RAF，下一帧统一更新一次。

```javascript
onTouchMove(e) {
    // 只存坐标
    this._pendingX = e.touches[0].clientX
    this._pendingY = e.touches[0].clientY
    // 同一帧内只请求一次 RAF
    if (!this._rafId) {
        this._rafId = requestAnimationFrame(() => {
            this._rafId = null
            if (this.isDragging) {
                this.updateFromTouch(this._pendingX, this._pendingY)
            }
        })
    }
}
```

**关键细节**：

| 场景 | 行为 |
|------|------|
| touchstart | **立即** updateFromTouch（首次触摸要即时反馈） |
| touchmove | 存坐标 + RAF 合帧（每帧最多一次 DOM 写入） |
| touchend | 取消 pending RAF → 用最后坐标做最终更新 → 回调逻辑层 |

---

### 优化 6：环形命中判断（旋钮手感）

**问题**：触摸事件绑在根节点（400×400rpx），只排除了中心按钮，整个方形区域都能触发拖动。用户手指落在弧线外的空白处、方形边角处也会改值，缺少"旋钮手感"。

**修复**：touchstart 中加环形距离判断，只有触点落在弧线环带附近才允许拖动。

```javascript
// 弧线可见区域 mask: 50%~60% 半径
// 触摸容差放宽到 35%~80%
const dist = Math.sqrt(dx * dx + dy * dy)
const ratio = dist / radius
if (ratio < 0.35 || ratio > 0.80) return  // 太靠内或太靠外，忽略
```

| 触点位置 | ratio | 行为 |
|----------|-------|------|
| 内圈空洞附近 | < 0.35 | 忽略 |
| **弧线环带** | **0.35 ~ 0.80** | **允许拖动** |
| 方形边角 | > 0.80 | 忽略 |

**注意**：命中判断只在 touchstart 做一次。一旦进入拖动状态，touchmove 不再判断距离——手指滑到环带外面也能继续调节，符合旋钮操作直觉。

---

### 优化 7：外部传入值 clamp 兜底

**问题**：renderjs 的 `onDataChange` 中直接 `this.currentValue = newData.value`，如果父组件传入非法值（小于 min、大于 max、NaN、undefined），视图层进入异常状态。

**修复**：统一做 Number 转换 + clamp。

```javascript
const val = Number(newData.value)
this.currentValue = isNaN(val) ? this.min : Math.max(this.min, Math.min(val, this.max))
```

**注意赋值顺序**：先赋 `min`/`max`，再 clamp `value`，否则 clamp 用的是旧的边界值。

---

### 总结

| # | 优化项 | 类型 | 影响 |
|---|--------|------|------|
| 1 | 圆心偏差修复 | Bug 修复 | 角度映射更精准 |
| 2 | CSS 变量驱动 | 性能 | 不拼字符串、不重建整段背景 |
| 3 | filter 移入 CSS | 性能 | 消除高频无意义样式写入 |
| 4 | DOM 引用缓存 | 性能 | 消除高频 querySelector |
| 5 | RAF 合帧 | 性能 | DOM 写入限制为每帧一次 |
| 6 | 环形命中判断 | 交互体验 | 旋钮手感，防误触 |
| 7 | 输入值 clamp | 防御性编程 | 抗脏数据 |

## PowerDial 小圆点指示器：Rotator 方案

**日期**：2026-03-23  
**涉及文件**：`components/ui-box/power-dial.vue`

---

### 需求

在 PowerDial 弧线轨道上显示一个跟随进度的小圆点，提供直观的力量刻度指示。

---

### 方案选型

有两种实现思路：

| 方案 | 做法 | 每帧 JS 操作 |
|------|------|--------------|
| **A. 笛卡尔坐标** | 每帧算 `sin/cos`，写 `left` + `top` | 2 次三角函数 + 2 次 style 写入 |
| **B. Rotator 旋转** | dot 固定位置不动，只改外层容器 `rotate()` | **1 次** style 写入（仅 `transform`） |

选择 **方案 B**。

---

### Rotator 方案精髓

核心思想：**dot 自己不动，只改 rotator 的 `rotate()` 就能让 dot 绕圆走**。

#### DOM 结构

```html
<!-- 外层旋转器：尺寸和位置与弧线元素完全一致 -->
<view class="dial-indicator-rotator">
    <!-- 小圆点：固定在 rotator 的 12 点钟方向 -->
    <view class="dial-indicator-dot"></view>
</view>
```

#### 坐标系拆解

```
  rotator（400×400rpx 圆形容器，top: 20rpx，与弧线重合）
  ┌──────────────────────────────────┐
  │          ● dot (top: 45rpx)      │  ← 12 点钟方向，弧线中线处
  │                                  │
  │            ○ 圆心                │
  │                                  │
  └──────────────────────────────────┘
```

dot 通过 `top: 45rpx; left: 50%; transform: translate(-50%, -50%)` 固定在弧线中线的 12 点钟位置。这个位置**永远不变**。

当 rotator 旋转时，dot 就像"粘在表盘上的刻度点"一样绕圆心走：

```
rotator rotate(-120°)  →  dot 出现在弧线起点（8 点钟方向）
rotator rotate(0°)     →  dot 在 12 点钟（弧线正中间）
rotator rotate(+120°)  →  dot 出现在弧线终点（4 点钟方向）
```

#### dot 的 top 值怎么来的？

```
mask: radial-gradient(circle, transparent 50%, black 50%, black 60%, transparent 60%)
```

`circle` 关键字默认以 farthest-corner 为半径。对 200×200rpx（弧线元素半宽半高）的容器：

$$R = \sqrt{200^2 + 200^2} \approx 282.8 \text{rpx}$$

弧线可见区域 50%~60%，中线 55%：

$$r_{mid} = 282.8 \times 0.55 \approx 155 \text{rpx}$$

dot 在 12 点钟方向，即圆心正上方 155rpx 处。圆心在容器 50%（200rpx），所以：

$$top = 200 - 155 = 45 \text{rpx}$$

---

### JS 侧：极简的 DOM 操作

renderjs 中，dot 的更新只有**一行**——和弧线进度更新写在一起：

```javascript
// updateDOM — 拖动时每帧调用
updateDOM(ratio) {
    const angle = ratio * this.totalAngle   // totalAngle = 240

    // 弧线进度：改 CSS 变量
    this.arcProgressEl.style.setProperty('--progress-angle', angle + 'deg')

    // 小圆点：改 rotator 的旋转角（起始 -120°）
    this.indicatorEl.style.transform = 'rotate(' + (this.startAngle + angle) + 'deg)'

    // kg 文字
    if (this.valueEl) this.valueEl.textContent = this.currentValue
}
```

`syncView()` 同理，额外处理开/关机时的 `opacity` 切换。

---

### 为什么这个方案对 renderjs 性能最友好？

在 uni-app 的 renderjs 架构中，**渲染层 JS 直接操作 DOM 是最快路径**——绕过了 Vue 响应式和跨线程通信。Rotator 方案完美利用了这一点：

| 对比维度 | 笛卡尔方案 | Rotator 方案 |
|----------|-----------|-------------|
| 每帧三角函数 | 2 次（sin + cos） | **0 次** |
| 每帧 style 写入 | 2 次（left + top） | **1 次**（transform） |
| 触发布局重排（reflow） | ✅ `left`/`top` 改变几何位置 | ❌ `transform` 走 GPU 合成层 |
| 代码复杂度 | 需维护圆心坐标和半径 | 一行 `rotate()` |

关键点：

1. **零三角函数**：dot 的圆轨道运动完全由 CSS `transform: rotate()` 完成，浏览器内部用 GPU 矩阵变换处理，比 JS 手算 sin/cos 快一个数量级。

2. **零 reflow**：`transform` 属于合成层属性（composite property），修改它**不会触发布局重排和重绘**，浏览器直接在 GPU 层面旋转。相比之下，修改 `left`/`top` 会引发 reflow → repaint → composite 全链路。

3. **与 RAF 合帧协同**：touchmove 的 RAF 合帧（优化 5）已经把 DOM 写入控制在每帧一次。Rotator 方案让这"每帧一次"的写入成本降到最低——仅一个 `transform` 赋值。

4. **与 DOM 缓存协同**：`indicatorEl` 在 `bindEvents()` 时一次性缓存，拖动全程零 querySelector。

**结论**：Rotator 方案的增量成本 ≈ 一次字符串拼接 + 一次 `style.transform` 赋值，对 renderjs 的高频循环几乎零负担。

## nvue 动作训练页：全屏切换与浮层渲染问题复盘

本次排查对象为 `pages/freeTraining/action-training-b.nvue`。问题表面上看是“小窗/全屏切换异常”和“返回按钮显示异常”，但实际涉及的是 `nvue + Android + 原生 video + 浮层控件` 的一组组合坑。

### 1. `nvue` 中不要默认按 WebView 经验理解层级与渲染

普通 `.vue` 页面和 `.nvue` 页面不是同一种渲染体系。前者更多是 WebView 逻辑，后者是原生渲染。很多在 `.vue` 页面里理所当然的行为，在 `nvue` 里并不稳定，包括：

- 原生 `video` 的层级行为
- 浮层和固定定位元素的可见性
- `z-index` 调整后的事件接收顺序
- `image` / SVG 资源在特定层里的显示结果

**经验**：只要页面是 `nvue`，就不要直接复用 Web 页面里的直觉判断。

### 2. 原生 `video` 的样式切换要用“整对象替换”

小窗切全屏时，双击事件其实是正常触发的，问题在于原生 `video` 没有稳定响应字段级 `style` 更新。  
最终验证下来的可行方案是：

- 不要只改 `videoStyleObj.top / left / width / height`
- 要直接整对象替换 `videoStyleObj = { ... }`
- 触摸层 `videoTouchStyleObj` 也同步整对象替换

**经验**：`nvue` 原生 `video` 对对象内部字段变更不可靠，全屏/小窗切换优先使用整对象替换。

### 3. `.vue` 封装组件不能想当然地塞进 `nvue`

项目里已经有封装好的 `components/ui-box/common-back-button.vue`，在普通 `.vue` 页面中工作正常。但将它直接放入 `action-training-b.nvue` 的固定顶部浮层后，标题可以显示，返回按钮却始终不稳定。

后来验证发现：

- 同位置的纯 `view` 自绘按钮可以显示
- 同位置的 `.vue` 封装返回按钮不稳定
- 同位置直接放 SVG / `image` 也可能失效

这说明问题不在资源路径，而更像是：

- `.vue` 组件跨到 `nvue` 顶层浮层的兼容性不足
- `image` / SVG 在该层里的原生渲染不稳定

**经验**：`nvue` 页面里的关键浮层控件，优先使用 `nvue` 页面内直接实现，不要默认复用普通 `.vue` 封装组件。

### 4. SVG / `image` 不显示，不一定是资源错了

这次返回按钮还有一个很典型的假象：

- SVG 文件本身存在
- 在普通 `.vue` 页面里能正常显示
- 但在 `nvue` 固定浮层中不显示

这类问题不能只从“路径对不对”去排查。更合理的判断顺序应该是：

1. 先验证同位置的纯色块或纯 `view` 是否显示
2. 若纯 `view` 能显示，而 `image` / SVG 不能显示，则优先怀疑原生层渲染兼容性
3. 不要继续在同一个不稳定点上反复试错，应该切到 `nvue` 原生可控实现

**经验**：在 `nvue` 的 Android 浮层里，SVG 不显示很可能是渲染链路问题，不是资源缺失。

### 5. `z-index` 和布局重构会影响事件命中

为了做“小窗/全屏无感切换”，曾尝试把两套控件布局合并成一个共享层。这个方向在交互设计上是对的，但在 `nvue` 实现里带来了副作用：

- 双击切全屏开始失效
- 原本正常的事件链路被打乱
- 说明新的层级关系改变了触摸命中顺序

恢复回 `v-if / v-else` 的两层结构后，双击切换又恢复正常。这说明问题不在双击判断逻辑本身，而在事件接收层的结构变化。

**经验**：

- `nvue` 页面里，`z-index`、浮层顺序、事件接收层结构是强耦合的
- 改布局时，不能只看视觉结果，还要警惕手势命中是否被新的层挡掉
- 如果原结构已经验证过“事件稳定”，重构时应尽量保留该结构

### 6. 当前收敛结论

这次最终收敛出来的几条硬规则：

- `nvue` 原生 `video` 切换小窗/全屏时，使用整对象替换样式
- `nvue` 的关键浮层控件优先页面内实现，不默认复用 `.vue` 组件
- `SVG / image` 在 `nvue` Android 固定浮层里不稳定时，直接切回纯 `view` 方案
- 调整 `z-index` 或控件层结构后，要重新验证双击、点击、拖动等事件是否仍然命中
- `nvue` 交互问题必须以真机 Android 验证为准，不能只看代码推理
