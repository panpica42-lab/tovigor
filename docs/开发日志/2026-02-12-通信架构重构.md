## 一、今天做了什么

### 1. 通信架构从 v2 升级到 v3

**问题**：v2 架构在 App 启动时就开始永久轮询，导致：
- 控制台报错 `Failed to execute the callback function`
- 资源浪费（首页、课程列表等页面无意义轮询）
- 回调 ID 失效（页面切换时）

**根因**：误以为下位机是"主动上报"模式，实际是"请求-响应"模式——不发命令就没有响应。

**解决方案**：v3 架构——读发分离，按需启停

```
App 启动 → connect() 只连接，不轮询
                ↓
用户开启力量开关 → startWorking() 启动发送(5Hz)+读取(50ms)
                ↓
用户关闭/离开页面 → stopWorking() + sendOnce(OFF)
```

### 2. serialService.js 新增 API

| API | 用途 |
|-----|------|
| `startWorking(force, mode, interval)` | 启动工作状态（发送+读取并行） |
| `stopWorking()` | 停止工作状态 |
| `updateWorkingForce(force)` | 更新发送的力量值（不重启定时器） |
| `sendOnce(force, mode)` | 单次发送（用于关闭时发送 OFF） |
| `isWorking()` | 查询是否在工作状态 |

## 五、关键概念速查

### 下位机通信模型
- **请求-响应式**：App 发 D180 → 下位机回 A9
- **不发命令就没有响应**

### D180 下行帧（App → 设备）
- 9 字节
- 包含：force（力量值）、forceMode（模式）
- 模式：OFF=0, CONST=1（恒力）

### A9 上行帧（设备 → App）
- 96 字节
- 包含：瞬时功率、卡路里、心率等

### 关闭力量的正确方式
```javascript
serialService.sendOnce(5, FORCE_MODE.OFF)  // force=5, mode=0
```
**注意**：`force=0 + mode=0` 是紧急停车，不要使用！
