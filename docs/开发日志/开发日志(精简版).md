## 2026-03-17

### Passive Event Listener 警告深度分析

**问题**：Android 真机调试时控制台频繁出现警告，污染日志输出
```
Unable to preventDefault inside passive event listener due to target being treated as passive.
```

**警告触发链条**：
```
内容超出容器 → 滑动触发滚动 → uni-app 框架调用 preventDefault() → 与 passive 模式冲突 → 警告
```

**原因分析**：
1. Android WebView 默认将 `touchstart`、`touchmove` 等事件设为 **passive 模式**（优化滚动性能）
2. passive 模式下，浏览器假定监听器不会调用 `preventDefault()`，可立即开始滚动
3. 但 uni-app 框架底层为实现「局部滚动」，会调用 `preventDefault()` 阻止页面整体滚动
4. 两者冲突 → 警告

**关键发现**：
| 场景 | 是否警告 | 原因 |
|------|---------|------|
| 普通 view 上滑动 | ❌ 无 | 没有滚动发生，无需 preventDefault |
| scroll-view 上滑动，内容未溢出 | ❌ 无 | 无实际滚动发生 |
| scroll-view 上滑动，内容溢出 | ⚠️ 有 | 触发滚动，框架调用 preventDefault |
| view + overflow:auto，内容溢出 | ⚠️ 有 | 同上，与是否用 scroll-view 无关 |

**尝试过的无效方案**：
- ❌ `main.js` 覆盖 `addEventListener` 强制关闭 passive —— uni-app 框架在此之前已注册监听器
- ❌ `@touchmove.stop.prevent` —— 丑陋的 hack，可能影响正常滚动

**最终结论**：
- 这是 **uni-app + Android WebView 的底层行为**，无法从应用层消除
- 警告**不影响任何功能**，只是控制台提示
- 正式发布的 release 包中用户看不到

**我们能做的**：
1. ✅ 不需要滚动的地方，不用 scroll-view（减少无意义的警告）
2. ✅ 需要滚动的地方，接受警告存在
3. ✅ HBuilderX 控制台使用过滤功能屏蔽该警告
    
    有效方案：
    不需要滚动 → 不用 scroll-view
    不需要滚动但仍有警告 → touch-action: none
    需要滚动 → 接受警告，用控制台过滤

**额外技巧：touch-action: none**

对于**不需要滚动**的区域，如果仍然出现警告，可以在 CSS 中添加：
```css
.no-scroll-area {
  touch-action: none;
}
```

**原理**：
- `touch-action` 告诉浏览器该元素如何处理触摸手势
- `none` = 不处理任何触摸手势（禁用滚动、缩放等）
- 浏览器不注册触摸事件监听器 → 没有 passive 冲突 → 无警告

**适用场景**：
| 场景 | 是否适用 |
|------|---------|
| 纯展示区域（不需要滑动） | ✅ 适用 |
| 按钮、开关等交互元素 | ✅ 适用 |
| 需要滚动的列表 | ❌ 不适用（会禁用滚动） |
| 需要双指缩放的区域 | ❌ 不适用（会禁用缩放） |

**注意**：`touch-action: none` 会完全禁用该元素的触摸手势，仅用于确实不需要滚动的区域。

### 踩坑记录：flex 容器中图片不显示

**场景**：力量评估页的人体模型图片设置 `height: 100%` 后不显示

**原因**：
- 父元素 `.body-model-area` 使用 `flex: 1`，高度是弹性计算的，没有明确像素值
- 子元素 `<image>` 设置 `height: 100%` 时，找不到确定的参照值，计算结果为 0
- `width: auto` 依赖 height 计算，height 为 0 则 width 也为 0，图片消失

**解决方案**：改用固定尺寸 `width: 500rpx; height: 900rpx;`，不依赖父元素高度

### 规则总结：flex 容器中的百分比高度

**不是绝对不能用，而是有条件：**

| 父元素高度 | 子元素 `height: 100%` | 结果 |
|-----------|----------------------|------|
| `height: 500px` / `100vh` / 固定rpx  |  有参照值 | 生效 |
| `height: auto`（内容撑开）            |  无参照值 | 为 0 |
| `flex: 1`（弹性计算）                 |  无参照值 | 为 0 |
| `flex: 1` + `height: 0`              |  强制基准 | 生效（hack） |

**问题链条示例**：
```
.page (height: 100vh)              ✓ 明确
  └── .content (flex: 1)           ✗ 弹性，无明确值
        └── .body-model-area (flex: 1)   ✗ 弹性，无明确值
              └── .body-image (height: 100%)  → 找不到参照！
```

**安全做法**：
1. **图片/固定内容**：直接用 `rpx` 固定尺寸
2. **需要撑满的容器**：用 `position: absolute; top:0; bottom:0;`
3. **flex 子项要用百分比**：给父元素加 `height: 0`（强制基准）

**排查口诀**：flex 容器里的东西不显示 → 先检查百分比高度链条

## 2026-03-12

### Flappy Bird 小游戏开发

#### 踩坑记录：管道和地面对不上

**场景**：下管道（pipe-bottom）设置 `bottom: 0` 后，与地面之间有明显空隙

**原因分析**：
```
布局结构：
├── bg-ground (position: absolute; bottom: 0; height: 20%)  ← 绝对定位在屏幕底部
├── nav-bar (文档流)
├── timer-display (文档流)
└── game-area (position: relative; height: H * 0.8)  ← 在文档流中，从 nav 下方开始
```

问题链条：
1. `game-area` **不是绝对定位**，它在 `nav-bar` + `timer-display` 下方的文档流中
2. `gameAreaHeight = 屏幕高度 * 0.8`
3. `game-area` 的实际底部 = nav高度 + timer高度 + H*0.8 = **超出了地面顶部**
4. 管道基于 `game-area` 定位，`bottom: 0` 对齐的是 `game-area` 底部，而不是地面顶部

**解决方案**：
```css
/* game-area 改为绝对定位，精确对齐地面 */
.game-area {
  position: absolute;
  top: 0;
  left: 0;
  right: 0;
  bottom: 20%;  /* 地面占 20%，游戏区域到地面顶部 */
}
```

**规则**：涉及多层定位对齐时，确保参照系一致。文档流元素的位置受前序元素影响，绝对定位元素的位置只受定位父元素影响。

#### 性能优化

**1. requestAnimationFrame 替代 setInterval**

| 对比项 | setInterval | requestAnimationFrame |
|-------|-------------|----------------------|
| 刷新率 | 固定 16ms（约 60fps） | 自动跟随屏幕刷新率 |
| 后台行为 | 继续执行，浪费资源 | 自动暂停 |
| 动画流畅度 | 可能丢帧 | 与渲染同步，更流畅 |

```javascript
// 改前
gameLoop = setInterval(updateGame, 16)

// 改后
const gameLoopRAF = (currentTime) => {
  if (!gameStarted.value || gameOver.value) return
  
  if (lastFrameTime === 0) lastFrameTime = currentTime
  const deltaTime = currentTime - lastFrameTime
  
  if (deltaTime >= 16) {  // ~60fps
    lastFrameTime = currentTime
    updateGame()
  }
  
  gameLoop = requestAnimationFrame(gameLoopRAF)
}
gameLoop = requestAnimationFrame(gameLoopRAF)
```

**2. 减少响应式开销**

问题：`pipes` 用 `ref([])` 时，每次修改 `pipe.x` 都会触发 Vue 响应式追踪，每帧触发多次。

解决：改用 `shallowRef` + 手动 `triggerRef`

```javascript
// 改前
const pipes = ref([])
// 每次 pipe.x -= PIPE_SPEED 都触发响应式

// 改后
const pipes = shallowRef([])
// 循环内直接修改 pipe.x，不触发响应式
// 循环结束后手动触发一次更新
triggerRef(pipes)
```

效果：从每帧触发 N 次（N = 管道数量）减少到每帧触发 1 次。
