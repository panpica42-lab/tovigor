## 2026-03-17

### Passive Event Listener 警告深度分析

**问题**：Android 真机调试时控制台频繁出现警告，污染日志输出
```
Unable to preventDefault inside passive event listener due to target being treated as passive.
```

**警告触发链条**：
```
内容超出容器 → 滑动触发滚动 → uni-app 框架调用 preventDefault() → 与 passive 模式冲突 → 警告
```

**原因分析**：
1. Android WebView 默认将 `touchstart`、`touchmove` 等事件设为 **passive 模式**（优化滚动性能）
2. passive 模式下，浏览器假定监听器不会调用 `preventDefault()`，可立即开始滚动
3. 但 uni-app 框架底层为实现「局部滚动」，会调用 `preventDefault()` 阻止页面整体滚动
4. 两者冲突 → 警告

**关键发现**：
| 场景 | 是否警告 | 原因 |
|------|---------|------|
| 普通 view 上滑动 | ❌ 无 | 没有滚动发生，无需 preventDefault |
| scroll-view 上滑动，内容未溢出 | ❌ 无 | 无实际滚动发生 |
| scroll-view 上滑动，内容溢出 | ⚠️ 有 | 触发滚动，框架调用 preventDefault |
| view + overflow:auto，内容溢出 | ⚠️ 有 | 同上，与是否用 scroll-view 无关 |

**尝试过的无效方案**：
- ❌ `main.js` 覆盖 `addEventListener` 强制关闭 passive —— uni-app 框架在此之前已注册监听器
- ❌ `@touchmove.stop.prevent` —— 丑陋的 hack，可能影响正常滚动

**最终结论**：
- 这是 **uni-app + Android WebView 的底层行为**，无法从应用层消除
- 警告**不影响任何功能**，只是控制台提示
- 正式发布的 release 包中用户看不到

**我们能做的**：
1. ✅ 不需要滚动的地方，不用 scroll-view（减少无意义的警告）
2. ✅ 需要滚动的地方，接受警告存在
3. ✅ HBuilderX 控制台使用过滤功能屏蔽该警告
    
    有效方案：
    不需要滚动 → 不用 scroll-view
    不需要滚动但仍有警告 → touch-action: none
    需要滚动 → 接受警告，用控制台过滤

**额外技巧：touch-action: none**

对于**不需要滚动**的区域，如果仍然出现警告，可以在 CSS 中添加：
```css
.no-scroll-area {
  touch-action: none;
}
```

**原理**：
- `touch-action` 告诉浏览器该元素如何处理触摸手势
- `none` = 不处理任何触摸手势（禁用滚动、缩放等）
- 浏览器不注册触摸事件监听器 → 没有 passive 冲突 → 无警告

**适用场景**：
| 场景 | 是否适用 |
|------|---------|
| 纯展示区域（不需要滑动） | ✅ 适用 |
| 按钮、开关等交互元素 | ✅ 适用 |
| 需要滚动的列表 | ❌ 不适用（会禁用滚动） |
| 需要双指缩放的区域 | ❌ 不适用（会禁用缩放） |

**注意**：`touch-action: none` 会完全禁用该元素的触摸手势，仅用于确实不需要滚动的区域。

### 踩坑记录：flex 容器中图片不显示

**场景**：力量评估页的人体模型图片设置 `height: 100%` 后不显示

**原因**：
- 父元素 `.body-model-area` 使用 `flex: 1`，高度是弹性计算的，没有明确像素值
- 子元素 `<image>` 设置 `height: 100%` 时，找不到确定的参照值，计算结果为 0
- `width: auto` 依赖 height 计算，height 为 0 则 width 也为 0，图片消失

**解决方案**：改用固定尺寸 `width: 500rpx; height: 900rpx;`，不依赖父元素高度

### 规则总结：flex 容器中的百分比高度

**不是绝对不能用，而是有条件：**

| 父元素高度 | 子元素 `height: 100%` | 结果 |
|-----------|----------------------|------|
| `height: 500px` / `100vh` / 固定rpx  |  有参照值 | 生效 |
| `height: auto`（内容撑开）            |  无参照值 | 为 0 |
| `flex: 1`（弹性计算）                 |  无参照值 | 为 0 |
| `flex: 1` + `height: 0`              |  强制基准 | 生效（hack） |

**问题链条示例**：
```
.page (height: 100vh)              ✓ 明确
  └── .content (flex: 1)           ✗ 弹性，无明确值
        └── .body-model-area (flex: 1)   ✗ 弹性，无明确值
              └── .body-image (height: 100%)  → 找不到参照！
```

**安全做法**：
1. **图片/固定内容**：直接用 `rpx` 固定尺寸
2. **需要撑满的容器**：用 `position: absolute; top:0; bottom:0;`
3. **flex 子项要用百分比**：给父元素加 `height: 0`（强制基准）

**排查口诀**：flex 容器里的东西不显示 → 先检查百分比高度链条

## 2026-03-12

### Flappy Bird 小游戏开发

#### 踩坑记录：管道和地面对不上

**场景**：下管道（pipe-bottom）设置 `bottom: 0` 后，与地面之间有明显空隙

**原因分析**：
```
布局结构：
├── bg-ground (position: absolute; bottom: 0; height: 20%)  ← 绝对定位在屏幕底部
├── nav-bar (文档流)
├── timer-display (文档流)
└── game-area (position: relative; height: H * 0.8)  ← 在文档流中，从 nav 下方开始
```

问题链条：
1. `game-area` **不是绝对定位**，它在 `nav-bar` + `timer-display` 下方的文档流中
2. `gameAreaHeight = 屏幕高度 * 0.8`
3. `game-area` 的实际底部 = nav高度 + timer高度 + H*0.8 = **超出了地面顶部**
4. 管道基于 `game-area` 定位，`bottom: 0` 对齐的是 `game-area` 底部，而不是地面顶部

**解决方案**：
```css
/* game-area 改为绝对定位，精确对齐地面 */
.game-area {
  position: absolute;
  top: 0;
  left: 0;
  right: 0;
  bottom: 20%;  /* 地面占 20%，游戏区域到地面顶部 */
}
```

**规则**：涉及多层定位对齐时，确保参照系一致。文档流元素的位置受前序元素影响，绝对定位元素的位置只受定位父元素影响。

#### 性能优化

**1. requestAnimationFrame 替代 setInterval**

| 对比项 | setInterval | requestAnimationFrame |
|-------|-------------|----------------------|
| 刷新率 | 固定 16ms（约 60fps） | 自动跟随屏幕刷新率 |
| 后台行为 | 继续执行，浪费资源 | 自动暂停 |
| 动画流畅度 | 可能丢帧 | 与渲染同步，更流畅 |

```javascript
// 改前
gameLoop = setInterval(updateGame, 16)

// 改后
const gameLoopRAF = (currentTime) => {
  if (!gameStarted.value || gameOver.value) return
  
  if (lastFrameTime === 0) lastFrameTime = currentTime
  const deltaTime = currentTime - lastFrameTime
  
  if (deltaTime >= 16) {  // ~60fps
    lastFrameTime = currentTime
    updateGame()
  }
  
  gameLoop = requestAnimationFrame(gameLoopRAF)
}
gameLoop = requestAnimationFrame(gameLoopRAF)
```

**2. 减少响应式开销**

问题：`pipes` 用 `ref([])` 时，每次修改 `pipe.x` 都会触发 Vue 响应式追踪，每帧触发多次。

解决：改用 `shallowRef` + 手动 `triggerRef`

```javascript
// 改前
const pipes = ref([])
// 每次 pipe.x -= PIPE_SPEED 都触发响应式

// 改后
const pipes = shallowRef([])
// 循环内直接修改 pipe.x，不触发响应式
// 循环结束后手动触发一次更新
triggerRef(pipes)
```

效果：从每帧触发 N 次（N = 管道数量）减少到每帧触发 1 次。

### serialService.js 增强：updateWorkingForce 支持模式切换

**问题**：`formal-training.vue` 只用恒力模式不需要运行时切模式，所以 `updateWorkingForce(force)` 只接受力量参数。自由训练页需要在开机状态下切换模式，现有 API 无法覆盖。

**改动**：给 `updateWorkingForce` 增加可选的第二参数 `forceMode`。

```javascript
// 改前
export function updateWorkingForce(force) {
  workingForce = force
}

// 改后
export function updateWorkingForce(force, forceMode) {
  workingForce = force
  if (forceMode !== undefined) {
    workingMode = forceMode
  }
}
```

**向后兼容**：`forceMode` 不传时只更新力量，行为与改前完全一致。现有调用方（`formal-training.vue` × 3处、`strength-assess.vue` × 1处）均不受影响。

**数据流分析**（updateWorkingForce → D180 下行帧）：

```
updateWorkingForce(force, forceMode)
        │
        ▼
  workingForce = force      ← 更新内存变量
  workingMode = forceMode   ← 可选更新
        │
        │  ... 等到下一个定时器 tick（≤200ms）...
        │
        ▼
  sendTimer 回调 → buildD180Frame({ force: workingForce, forceMode: workingMode })
        │
        ▼
  D180 帧（9字节）：
  [0]帧头 [1]0xB4 [2]seq [3]force [4]forceMode [5]lock [6-7]CRC [8]帧尾
                          ~~~~~~~~  ~~~~~~~~~~
                          字节[3]    字节[4] ← 这两个字段受影响
```

**具体示例**：用户在 30kg 恒力模式下切到弹力模式

| 时刻 | 操作 | workingForce | workingMode | D180 字节[3][4] |
|------|------|---|---|---|
| T0 | 定时器 tick | 30 | 1(恒力) | `1E 01` |
| T1 | `updateWorkingForce(30, 4)` | 30 | **4(弹力)** | — |
| T2 | 下一个 tick | 30 | 4 | `1E 04` ✅ |

| 文件 | 改动 |
|------|------|
| `utils/serialService.js` | `updateWorkingForce` 增加可选 `forceMode` 参数 |
| `pages/freeTraining/free-training.vue` | 导入 serialService + FORCE_MODE，接入通信 |

## 2026-03-24

### 1. 训练时间计时重构：硬件 sportTime → JS 端速度检测计时

**问题**：看板训练时间使用的是 A9 帧的 `p.sportTime`（偏移 [35..38]），这是设备运行时长——开机就走，不运动也在涨，不符合"训练时间"语义。

**方案**：JS 端自己计时，基于"最后活跃时间戳 + 空闲窗口"模型：

```
A9 帧到达 (5Hz)
  └─ |left.speed| > SPEED_THRESHOLD 或 |right.speed| > SPEED_THRESHOLD ?
       └─ 是 → lastActiveTs = Date.now()

1 秒定时器
  └─ now - lastActiveTs < IDLE_WINDOW ?
       ├─ 是 → activeSeconds++（还在运动窗口内）
       └─ 否 → 跳过（空闲超过阈值，不计时）
```

**检测字段选型**：

| 候选字段 | 结论 | 理由 |
|----------|------|------|
| **speed** ✅ | 采用 | 最直觉（绳动=在练）；6 种动态模式全覆盖；抗误判（静止时≈0） |
| instantForce | 否决 | 握着绳不动也有读数（手臂自重+预紧力），假阳性高 |
| power | 否决 | 慢速大力动作（离心慢放）功率低，容易误判空闲 |
| sportCount 变化 | 否决 | 一个 rep 2-3 秒，期间 count 不变，配合窗口会"明明在练但不计时" |

**注意**：协议中 speed 是有符号 int16（可为负值），判断时取 `Math.abs()`.

**新增可调常量**（均为普通变量，方便现场调试）：

| 常量 | 默认值 | 说明 |
|------|--------|------|
| `SPEED_THRESHOLD` | 10 mm/s | 任一手速度绝对值 > 此值视为在运动 |
| `IDLE_WINDOW` | 2000 ms | 超过此时长无活跃帧则暂停计时 |

**计时器生命周期**：开机时 `setInterval`，关机时 `clearInterval`，`onBeforeUnmount` 兜底清理。`activeSeconds` 跨组累加，页面销毁归零。

### 2. 关机发送序列：单次 → 持续 3 秒

**问题**：关机只发一帧 OFF 指令，如果那一帧丢了（串口无确认机制），下位机不知道要关闭。

**方案**：关机时以工作同频（5Hz）持续发送 OFF 指令 3 秒：

```
用户点关机
  ├─ 立即发一帧 sendOnce(5, FORCE_MODE.OFF)  ← 不等首个 interval
  ├─ setInterval 每 200ms 再发一帧
  └─ setTimeout 3 秒后自动停止
  总计 ≈ 16 帧 OFF 指令
```

**三个清理入口**（保证不泄漏定时器）：

| 场景 | 清理方 |
|------|--------|
| 3 秒自然到期 | `shutdownTimeout` 回调 → `stopShutdownSequence()` |
| 3 秒内重新开机 | ON 分支开头 → `stopShutdownSequence()` |
| 3 秒内退出页面 | `onBeforeUnmount` → `stopShutdownSequence()` |

**新增可调常量**：

| 常量 | 默认值 | 说明 |
|------|--------|------|
| `SHUTDOWN_DURATION` | 3000 ms | 关机指令持续发送时长 |
| `SHUTDOWN_INTERVAL` | 200 ms | 关机指令发送频率 |

## 2026-03-25 动作训练页视频小窗功能开发

### 目标

在 `action-training.vue` 页面实现视频播放功能：
- 进入页面弹出安全须知弹窗，点击确认后视频在小窗中自动播放
- 双击小窗 → 视频变成全屏背景（力量旋钮、看板等控件叠在上层）
- 双击全屏 → 回到小窗模式

### 踩坑记录

#### 坑1：`getBoundingClientRect` of null

**现象**：每次进入页面就报 `Uncaught TypeError: Cannot read properties of null (reading 'getBoundingClientRect')`

**原因**：`showSafetyModal` 初始值为 `false`，模板中 `v-if="!showSafetyModal"` 导致 video 在首帧就被创建。uni-app 的 `<video>` 是原生组件，视图层内部需要 `getBoundingClientRect` 定位覆盖层，但此时 DOM 尚未就绪。`onMounted` 才把 `showSafetyModal` 改为 `true`，video 又被销毁——先建后拆，白白触发了错误。

**修复**：把 `showSafetyModal` 初始值改为 `true`，首帧就不创建 video，从源头消除问题。

#### 坑2：两个 video 互斥切换导致播放中断

**现象**：最初方案是两个 `v-if` 互斥的 video（一个小窗、一个全屏），切换时视频会中断重新加载。

**修复**：改为只保留一个 video 实例，通过动态 `:class` 切换容器样式（`floating-video-window` ↔ `video-bg-layer`），避免销毁/重建。

#### 坑3：视频不能自动播放

**现象**：弹窗确认后，小窗黑屏显示播放按钮，需要手动点击才能播放。`:autoplay="canAutoPlay"` 动态绑定无效。

**修复**：video 标签写死 `autoplay`，同时在 `handleModalConfirm` 中用 `setTimeout` + `uni.createVideoContext('actionVideo').play()` 主动调用播放。

#### 坑4：普通 `<view>` 无法拦截原生 video 的触摸事件

**现象**：在 video 上方覆盖了 `<view>` 触摸层，但点击仍然穿透到 video，触发了默认的暂停/播放行为，双击切换全屏也没反应。

**原因**：uni-app 的 `<video>` 是原生组件，渲染在独立的 Native 层，普通 `<view>` 无法覆盖。

**修复**：改用 `<cover-view>` 作为触摸拦截层，这是 uni-app 专门为覆盖原生组件设计的。

#### 坑5：`cover-view` 上使用 `.stop` 修饰符报错

**现象**：`@tap.stop="handleVideoTap"` 在 Android App 端报 `e.stopPropagation is not a function`。

**原因**：`cover-view` 的事件对象不是标准 DOM Event，没有 `stopPropagation` 方法。

**修复**：去掉 `.stop` 修饰符，改为 `@tap="handleVideoTap"`。

#### 坑6：全屏模式下 video 覆盖所有控件

**现象**：双击进入全屏后，视频确实铺满了屏幕，但力量旋钮、看板等全部被遮住，无法操作。

**原因**：Android 端的原生 `<video>` 组件永远渲染在 WebView 之上，CSS `z-index` 对它无效。普通 `<view>` 不可能盖住它。

**最终方案**：接受原生组件的层级限制，全屏模式下用 `cover-view` 叠加简化的信息层（顶部：返回按钮 + 动作名，底部：关键训练数据，中间：「双击退出全屏」提示）。力量旋钮等复杂交互只在小窗模式下使用。

### 关键经验总结

| 知识点 | 说明 |
|--------|------|
| 原生组件层级 | uni-app Android 端的 `<video>` / `<map>` / `<camera>` 渲染在 Native 层，永远在 WebView 之上 |
| `cover-view` | 唯一能覆盖原生组件的方案，但功能受限（不支持复杂样式、事件对象非标准 DOM Event） |
| `cover-view` 事件 | 不能用 `.stop` / `.prevent` 等 Vue 修饰符，事件对象缺少标准方法 |
| video 自动播放 | 动态 `:autoplay` 不可靠，应该用 `uni.createVideoContext().play()` 主动触发 |
| `v-if` + 原生组件 | 首帧就创建又立刻销毁原生组件容易触发 `getBoundingClientRect` 空指针，初始状态要规划好 |
| 单实例 video | 避免两个 video 用 `v-if` 互斥切换（销毁重建会中断播放），改用 class 切换样式 |

## 2026-03-26 动作训练页方案B：nvue 重写 + 串口上下文隔离问题

### 背景：为什么从 vue 切换到 nvue？

昨天（3/25）在 `action-training.vue`（WebView 页面）中实现视频小窗功能，踩了 6 个坑，最终结论是：

> uni-app Android 端的 `<video>` 是原生组件，永远渲染在 WebView 之上，CSS `z-index` 对它无效。  
> `cover-view` 虽能覆盖原生组件，但功能严重受限——不支持复杂样式、不支持 `slider`、事件对象非标准。

这意味着在 vue 页面中，**全屏模式下力量 slider 等复杂控件无法叠加在视频上方**，只能用 `cover-view` 做简化信息展示，交互体验大打折扣。

**方案B（nvue）的优势**：

| 对比项 | vue 页面（WebView） | nvue 页面（Native 渲染） |
|--------|---------------------|--------------------------|
| video 层级 | Native 层，永远在最上 | 与其他组件同层 |
| z-index 控制 | 对 video 无效 | ✅ 有效，可自由叠加 |
| slider 覆盖 video | ❌ 不可能 | ✅ 可以 |
| cover-view 限制 | 必须用，样式阉割 | 不需要 |
| 全屏模式交互 | 只能展示简化信息 | ✅ 完整控件可用 |

因此决定新建 `action-training-b.nvue`，用 nvue 方案重写动作训练页。

### 页面架构设计

```
z-index 分层（nvue 支持）：
  小窗模式：sw-layout(3) < video(8) < video-touch(9)
  全屏模式：video(1) < video-touch(4) < fs-layout(7) < modal(20)

交互逻辑：
  小窗模式 → 视频悬浮右侧，控件正常布局
  双击视频 → 全屏（视频铺满，控件半透明叠加）
  双击中间空白 → 退出全屏
```

- 使用 Options API（nvue 对 Composition API 支持有限）
- 单例 video 实例，通过 computed 动态切换 style（小窗 ↔ 全屏）
- 原生 `<slider>` 组件控制力量值（nvue 同层渲染，可覆盖 video）

### 踩坑记录

#### 坑1：小窗视频宽高反转

**现象**：小窗视频显示为横向长条，预期是竖向窗口（宽 < 高）。

**原因**：计算公式写反了
```javascript
// ❌ 错误：横屏 16:9（宽 > 高）
const svW = Math.round(sw * 0.32)        // 横向=大值
const svH = Math.round(svW * 9 / 16)     // 纵向=小值

// ✅ 正确：竖屏 9:16（宽 < 高）
const svH = Math.round(sh * 0.40)        // 纵向=大值（屏幕高度 40%）
const svW = Math.round(svH * 9 / 16)     // 横向=小值
```

#### 坑2：`createVideoContext` 空指针

**现象**：视频偶尔无法播放，控制台可能报空指针。

**原因**（三个隐患叠加）：
1. nvue 中 `uni.createVideoContext(id)` 缺少第二参数 `this`，可能找不到组件
2. 多处（mounted、handleModalConfirm、handleVideoTap）各自重复写创建逻辑，无统一保护
3. 原生层未就绪时 `createVideoContext` 可能抛异常，无 try-catch

**修复**：抽取 `ensureVideoCtx()` 统一安全方法：
```javascript
ensureVideoCtx() {
    if (!this.videoCtx) {
        try {
            this.videoCtx = uni.createVideoContext('actionVideo', this) // 传 this
        } catch (e) {
            console.warn('[action-training-b] createVideoContext 失败:', e)
            return null
        }
    }
    return this.videoCtx
}
```

#### 坑3：nvue 不支持的 CSS 属性

**现象**：视频圆角无效、控制台报样式警告。

**原因**：
- nvue 原生 `<video>` 不支持 `border-radius`
- nvue 不支持 `overflow: hidden`

**涉及代码**：
```javascript
// ❌ videoStyle 中（已移除）
{ borderRadius: '10px', overflow: 'hidden' }

// ❌ .container CSS 中（已移除）
.container { overflow: hidden; }
```

**修复**：直接移除这些属性，nvue 中不使用圆角视频。

#### 坑4：视频资源泄漏

**现象**：页面卸载后视频仍在后台播放（耗电、占资源）。

**原因**：`beforeDestroy` 中只清理了串口和定时器，未停止视频。

**修复**：在 `beforeDestroy` 最前面添加：
```javascript
if (this.videoCtx) {
    try { this.videoCtx.pause() } catch (e) { /* ignore */ }
    this.videoCtx = null
}
```

#### 坑5：串口通信完全失效（最关键！）

**现象**：点击"开始训练"后，下位机无任何响应。点击"停止"时控制台打印 `[serialService] 发送跳过：未连接`。但在其他 `.vue` 页面（如 `free-training.vue`）串口工作正常。

**原因分析**：

> **nvue 页面有独立的 JS 上下文，与 vue 页面（WebView）不共享模块状态！**

```
┌─── WebView JS 上下文 ───────────────────────────────┐
│  App.vue → connect() → isConnected = true ✅         │
│  free-training.vue → startWorking() → 正常 ✅        │
│  （所有 .vue 页面共享这一份 serialService 实例）       │
└──────────────────────────────────────────────────────┘

┌─── Native JS 上下文（独立！）────────────────────────┐
│  action-training-b.nvue                              │
│  → import serialService（重新加载一份全新实例！）      │
│  → isConnected = false, portId = 0 ❌                │
│  → startWorking() → "发送跳过：未连接"                │
└──────────────────────────────────────────────────────┘
```

`serialService.js` 中的 `isConnected`、`portId` 等模块级变量，在两个 JS 上下文中各自独立。App.vue 的 `connect()` 只设置了 WebView 上下文中的状态，nvue 页面拿到的是一份全新的、未连接的实例。

**修复**：nvue 页面需要自行管理串口连接生命周期：

```javascript
// mounted 中
async ensureSerialConnection() {
    const status = serialService.getStatus()
    if (status.isConnected) return  // 已连接则跳过
    await serialService.connect({ path: '/dev/ttyS9', baudRate: 115200 })
}

// beforeDestroy 中
serialService.disconnect()  // 离开时清理此上下文的连接
```

### 关键经验总结

| 知识点 | 说明 |
|--------|------|
| nvue JS 上下文隔离 | **nvue 和 vue 页面运行在不同 JS 上下文中**，JS 模块的单例状态（闭包变量）不共享。这是最容易被忽视的坑 |
| nvue 需自行连接串口 | App.vue 中的 `connect()` 只影响 WebView 上下文，nvue 页面必须自己调用 `connect()` |
| nvue 不支持 overflow | `overflow: hidden` 在 nvue 中无效，不要使用 |
| nvue video 无圆角 | 原生 `<video>` 不支持 `border-radius`，圆角裁切无效 |
| createVideoContext + this | nvue 中必须传第二参数 `this`：`uni.createVideoContext('id', this)` |
| 视频资源回收 | `beforeDestroy` 中必须 `pause()` + 置空 `videoCtx`，防止后台持续播放 |

## 补充记录：nvue 视频小窗/全屏切换异常排查

### 现象

Android App 中，`pages/freeTraining/action-training-b.nvue` 的视频区域双击后“有反应”，但不是预期中的“小窗视频放大为全屏视频”。控件层会切换，视频本身却没有正确从小窗变为全屏。

### 初始误判

一开始容易怀疑是：

- 双击事件没有触发
- 触摸层没有覆盖到原生 video
- 循环播放或 `loadedmetadata + seek` 逻辑干扰了切换

但实际排查结果表明，双击事件本身是正常的，`isVideoFullscreen` 也发生了切换。

### 真正原因

问题出在 `video` 的样式更新方式。

当时为了避免切换全屏时重建原生 `video` 视图、尽量保留播放进度，`_applyVideoStyle()` 采用了“保留同一个对象引用，只修改 `videoStyleObj.top/left/width/height/zIndex` 字段”的写法。

这套写法在普通 Vue DOM 中通常没问题，但在 `nvue` 的原生 `video` 桥接层里并不可靠。结果是：

- Vue 状态变了
- 控件布局变了
- 但原生 `video` 没有稳定响应这类字段级 style 变更

于是表现成“看起来切换了，但视频尺寸没真正更新”。

### 修复

将 `_applyVideoStyle()` 从“原地修改对象字段”改为“整对象重新赋值”：

- `this.videoStyleObj = { ... }`
- `this.videoTouchStyleObj = { ... }`

实机验证后，视频可以正确从小窗切换到全屏，问题确认解决。

另外，排查过程中临时给触发区同时绑定了 `@tap` 和 `@click`，这会让一次点击被处理两次，导致看起来像“单击就能切换”。最终应只保留 `@tap`，恢复为真正的双击切换。

### 反思

这次问题的关键经验：

- `nvue` 原生组件不要默认按 Web DOM 的响应式行为去理解
- 对原生 `video` 这类桥接组件，`style` 的字段级变更不一定可靠
- 如果界面状态变了但原生视图没变，优先怀疑“原生层未吃到更新”，而不是先怀疑事件逻辑
- 为了“避免重建”而做的优化，必须先经过真机验证，否则很容易引入更隐蔽的问题

结论可以直接记住：

> 在 `nvue` 中，原生 `video` 的样式切换优先使用“整对象替换”，不要依赖“对象内部字段原地修改”。

## 2026-03-31 动作训练页 B（nvue）流程整理

- 重构了 `pages/freeTraining/action-training-b.nvue` 的训练流程状态，明确拆分为四个阶段：
  `idle`：空闲/待机态，尚未开始训练，或用户主动停止后回到待机。

- 重写了“下一个动作”倒计时逻辑，改为基于截止时间戳计算剩余秒数，而不是单纯依赖 `countdown--`，降低了定时器抖动和状态重入带来的显示异常风险。

- 修复了动作切换弹窗倒计时看起来卡住的问题。根因是串口帧持续上报时，页面会重复进入“当前动作完成”逻辑，导致倒计时被反复重置。现已改为只有 `training` 状态允许结算，进入 `transitioning` 后不再重复触发完成逻辑。

- 修复了数据看板“次数一上来显示负数”的问题。根因是设备当前 `sportCount` 小于页面保存的 `countBaseline` 时，直接做减法会得到负值。现已在检测到计数回退时立即重抓基线，并对看板显示做非负数兜底，避免出现 `-140` 这类异常次数。

- 补充了文件头部注释，说明了四个训练流程状态的含义，方便后续维护直接从页面顶部理解状态机设计。
