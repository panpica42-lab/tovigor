# wzl-serialbridge 串口通信插件

基于 UTS 的 Android 串口通信插件,支持 UART 设备的读写操作。

## 为什么需要 UTS 插件？

### 前端语言无法直接访问串口

在 uni-app 中，JavaScript/Vue 运行在受限的沙箱环境里，存在以下限制：

| 限制 | 说明 |
|------|------|
| 浏览器沙箱 | JS 运行在沙箱里，不能直接访问硬件 |
| 没有串口 API | uni-app 的 JS 引擎没有提供访问 /dev/ttyS3 的能力 |
| 权限问题 | 串口是 Linux 设备文件，需要系统级权限 |

### 调用链路

```
JS (前端)
   |
   | uni-app 插件机制
   v
UTS (桥接层，编译成 Kotlin)
   |
   | import AAR
   v
Kotlin AAR (原生层)
   |
   | Java FileInputStream / NDK
   v
Linux 内核  -->  /dev/ttyS3 (串口硬件)
```

## UTS 插件与 AAR 的关系

### 什么是 UTS？

UTS (Uni TypeScript) 是 uni-app 提供的跨平台原生插件开发语言，语法类似 TypeScript，但可以直接编译成各平台的原生代码：

| 平台 | UTS 编译目标 |
|------|-------------|
| Android | Kotlin |
| iOS | Swift |
| 鸿蒙 | ArkTS |

### 什么是 AAR？

AAR (Android Archive) 是打包好的 Android 原生库，包含 Kotlin/Java 编译后的字节码。

### 本插件的架构

```
serialbridge.aar (Kotlin)           index.uts (UTS)                  serial-test.vue (JS)
+--------------------+              +------------------+             +------------------+
| class SerialPort { |  import      | import SerialPort|   export    | import {         |
|   open()           | <----------  | from 'com...aar' | <---------- |   openSerial     |
|   read()           |              |                  |             | } from '...'     |
|   write()          |              | fun openSerial() |             |                  |
|   close()          |              |   SerialPort()   |             | openSerial({...})|
| }                  |              | }                |             |                  |
+--------------------+              +------------------+             +------------------+
     纯 Android 代码                   桥接/适配层                       业务代码
```

### 为什么需要 AAR？

UTS 虽然能写 Kotlin，但有些场景需要 AAR：

| 场景 | 说明 |
|------|------|
| 复用现有库 | 已有成熟的 Kotlin/Java 串口库，直接打包成 AAR 引入 |
| 复杂原生功能 | JNI、NDK、硬件驱动等，UTS 不方便写 |
| 保护代码 | AAR 是编译后的二进制，比 UTS 源码更难反编译 |
| 团队分工 | Android 开发写 AAR，前端开发写 UTS 对接 |

### 简单总结

- UTS = uni-app 的"翻译官"，让 JS 能调用原生代码
- AAR = 打包好的 Android 原生库（Kotlin/Java 编译产物）
- 关系：UTS 通过 import 语句导入 AAR 中的类，然后封装成 uni-app 风格的 API 供 JS 调用

## 架构演进历程

本项目的串口通信架构经历了三个版本的演进：

### v1：探索版（直接调用 UTS）

**目标**：验证 UTS 插件能否正常工作

**代码**：`pages/serial-test/serial-test.vue`（839 行）

**架构**：
```
页面 (serial-test.vue) ──直接调用──> UTS 插件 (wzl-serialbridge)
```

**特点**：
- 页面直接 import UTS 插件的 `openSerial`、`readSerial` 等函数
- 页面自己维护 `portId`、轮询定时器
- 无帧解析（TODO 注释提醒）
- 无拆包/粘包处理
- 无 CRC 校验

**问题**：
- 每个页面都要重复写连接管理代码
- 页面切换后串口状态丢失
- 帧解析逻辑无法复用

**定位**：保留作为**底层调试工具**，排查 UTS 层问题

---

### v2：服务层版（引入 serialService）

**目标**：抽象出服务层，实现单例管理和帧解析

**代码**：`pages/serial-test/serial-test-v2.vue`（766 行）+ `utils/serialService.js`

**架构**：
```
页面 (serial-test-v2.vue)
    │
    │ serial.connect() / serial.on('frame', callback)
    ▼
serialService.js（单例）
    │
    │ 帧解析 + CRC 校验
    ▼
UTS 插件 (wzl-serialbridge)
```

**特点**：
- serialService 单例管理连接
- 事件订阅模式（`on('frame', callback)`）
- 帧解析：拆包/粘包处理、CRC-16 校验
- 页面负担减轻，但**页面仍管理连接生命周期**

**问题**：
- 页面仍然调用 `connect()`/`disconnect()`
- 多页面场景下连接管理不统一

**定位**：过渡版本，验证服务层设计

---

### v3：生产版（App.vue 统一管理）⭐ 当前使用

**目标**：连接生命周期与应用生命周期绑定，页面只关注业务

**代码**：`App.vue` + 业务页面（如 `skip1.vue`）+ `utils/serialService.js`

**架构**：
```
App.vue（应用入口）
    │
    │ connect()    <-- 应用启动时
    │ disconnect() <-- 应用退出时
    ▼
┌─────────────────────────────────────────────┐
│  serialService.js（单例）                    │
│  - 连接管理                                  │
│  - 帧解析 + CRC                             │
│  - 事件分发                                  │
└─────────────────────────────────────────────┘
    ▲
    │ on('frame', handler)   <-- 订阅
    │ startWorking()         <-- 按钮触发
    │ stopWorking()          <-- 离开页面
    │
业务页面 (skip1.vue, free-training.vue ...)
```

**特点**：
- **App.vue 负责连接**：应用启动时 `connect()`，退出时 `disconnect()`
- **页面不调用 `connect()`**：只订阅事件 + 控制工作状态
- **按钮触发通信**：用户点击"开始"才 `startWorking()`
- 页面职责清晰：订阅 → 处理数据 → 更新 UI

**核心原则**：
```javascript
// ✅ 正确的页面代码（v3 架构）
import { on, off, startWorking, stopWorking, getStatus } from '@/utils/serialService.js'

onMounted(() => {
  on('frame', handleFrame)           // 订阅
  const status = getStatus()         // 检查（不是建立连接）
})

const handleStart = () => {
  startWorking(force, mode, 200)     // 用户点击才启动
}

onBeforeUnmount(() => {
  off('frame', handleFrame)
  stopWorking()                       // 停止，但不断开连接
})
```

## 推荐的分层架构设计

### 背景与问题

本项目在开发初期使用 Android 原生开发（Android Studio），当时的通信架构是：
- 串口通信对象放在长生命周期的组件（Service）里统一维护
- Activity 之间不直接"传串口数据"
- 把 Service 当成"通信中枢"：各个 Activity 绑定到同一个 Service

现在迁移到 uni-app Vue 框架后，需要重新设计通信架构。

### 当前临时方案的问题

当前 serial-test.vue 直接调用 UTS 插件：

```
serial-test.vue  ──直接调用──>  UTS 插件 (wzl-serialbridge)
```

这种方式存在问题：
- 每个页面都要自己管理 portId
- 每个页面都要自己写轮询逻辑
- 页面切换后串口状态丢失
- 帧解析逻辑（拆包/粘包）要到处复制

### 推荐的分层架构

```
+---------------------------------------------------------------+
|  Vue 页面层 (free-training.vue, serial-test.vue ...)           |
|  - UI 交互                                                     |
|  - 调用 serialService                                          |
+---------------------------------------------------------------+
                              |
                              v
+----------------------------------------------------------------+
|  JS 服务层 (serialService.js) <-- 新增的，纯 JS 文件            |
|  - 单例管理                                                    |
|  - 事件订阅/分发                                               |
|  - 帧解析(拆包/粘包)                                           |
|  - 调用 UTS 插件                                               |
+---------------------------------------------------------------+
                              |
                              v
+----------------------------------------------------------------+
|  UTS 插件层 (wzl-serialbridge) <-- 现有的，不用改！             |
|  - openSerial / closeSerial                                   |
|  - readSerial / writeSerial                                   |
|  - 调用 AAR                                                   |
+----------------------------------------------------------------+
                              |
                              v
+----------------------------------------------------------------+
|  AAR 层 (serialbridge.aar) <-- 现有的，不用改！                 |
|  - Kotlin 代码                                                 |
|  - 直接操作 /dev/ttyS*                                         |
+----------------------------------------------------------------+
```

### 各层职责划分

| 层 | 文件 | 职责 | 需要修改？ |
|---|------|------|----------|
| AAR | serialbridge.aar | 底层串口操作（打开、读、写、关闭） | 不用改 |
| UTS | wzl-serialbridge/*.uts | 把 AAR 暴露给 JS | 不用改 |
| JS 服务 | serialService.js | 单例+事件+帧解析 | 新增 |
| Vue 页面 | *.vue | UI 交互 | 改调用方式 |

### 功能应该放在哪一层？

| 功能 | 放哪里 | 原因 |
|------|--------|------|
| 打开/关闭串口 | AAR + UTS | 已有，不动 |
| 读/写字节 | AAR + UTS | 已有，不动 |
| HEX/Base64 转换 | UTS | 已有，不动 |
| 连接单例管理 | JS 服务层 | Vue 框架里更方便 |
| 帧解析(73...65) | JS 服务层 | 业务逻辑，纯 JS 就能做 |
| 事件订阅/分发 | JS 服务层 | 多页面共享数据 |
| 轮询定时器 | JS 服务层 | 集中管理 |
| UI 显示 | Vue 页面 | 展示层 |

### 关键结论

**不需要改 AAR，不需要改 UTS 插件！**

只需要在 JS 层加一个服务模块（serialService.js），它调用现有的 UTS 插件，然后提供更高级的封装：

```javascript
// serialService.js 核心思路

import { 
  openSerial,   // <-- 调用现有的 UTS 插件
  writeSerial, 
  readSerial, 
  closeSerial 
} from '@/uni_modules/wzl-serialbridge'

// 然后封装成更高级的 API
export function connect(path) {
  openSerial({ path, ... })  // 调用 UTS
}

export function send(data) {
  writeSerial({ ... })       // 调用 UTS
}

export function on(event, callback) {
  // 事件订阅机制
}
```

### 加了 JS 服务层之后的调用关系

```
serial-test.vue   ──调用──>  serialService.js  ──调用──>  UTS 插件
free-training.vue ─────────┘
其他页面 ─────────────────┘
```

好处：
- 串口连接全局唯一（单例）
- 轮询、帧解析只写一次
- 页面只需要订阅事件
- 页面切换不影响串口连接
- 关系：UTS 通过 import 语句导入 AAR 中的类，然后封装成 uni-app 风格的 API 供 JS 调用

### 数据流与事件设计

serialService.js 内部的数据流处理：

```
轮询 readSerial()
       |
       v
┌─────────────────────────────────────────────────────────┐
│  serialService.js                                       │
│                                                         │
│  每次读到数据 ──┬──> emit('raw', 原始数据)    ← 会拆包！ │
│                │                                        │
│                └──> rxBuffer 累积                       │
│                         │                               │
│                         v                               │
│                   检查帧头+长度+帧尾                    │
│                         │                               │
│                    完整帧？                             │
│                    /      \                             │
│                  是        否                           │
│                  │          │                           │
│                  v          v                           │
│        emit('frame', 完整帧)  继续等待                  │
│            ↓                                            │
│       只有完整帧！                                      │
└─────────────────────────────────────────────────────────┘
```

### 事件类型对比

| 事件 | 用途 | 是否拆包 | 适用场景 |
|------|------|---------|---------|
| `'raw'` | 调试用 - 观察底层读取情况 | 会拆包 | 串口调试页面 |
| `'frame'` | 业务用 - 获取解析后的完整帧 | 不会 | 正式业务页面 |
| `'connect'` | 连接成功通知 | - | 状态监听 |
| `'disconnect'` | 断开连接通知 | - | 状态监听 |
| `'error'` | 错误事件 | - | 异常处理 |

### 页面订阅示例

**调试页面（需要看原始数据）：**

```javascript
// serial-test-v2.vue
serial.on('raw', (data) => {
  // 每次轮询读到的原始碎片，可能不完整
  console.log('原始数据:', data.data, data.bytes, '字节')
})

serial.on('frame', (data) => {
  // 完整解析后的帧，永远是 96 字节
  console.log('完整帧:', data.parsed)
})
```

**正式业务页面（只关心完整帧）：**

```javascript
// free-training.vue
serial.on('frame', (data) => {
  // 只订阅 frame 事件，完全不需要关心拆包问题
  const force = data.parsed.left.instantForce
  const mode = data.parsed.setForceModeText
  
  // 更新 UI
  currentForce.value = force
  currentMode.value = mode
})
```

### 为什么保留 'raw' 事件？

虽然 `'raw'` 会显示拆包现象，但它有重要的调试价值：

1. **观察底层通信** - 确认数据是否正常到达
2. **排查硬件问题** - 如果 raw 没数据，说明是底层问题
3. **分析拆包规律** - 了解每次轮询读取的字节数
4. **对比帧解析效果** - 验证 frame 事件是否正确拼接

在正式业务页面中，完全忽略 `'raw'` 事件即可。

## 实践案例：定时器泄漏问题

### 问题背景

2026年2月9日，在开发串口测试页面时遇到一个典型 Bug：

**现象**：从 serial-test.vue（v1）页面跳转到其他页面后，HBuilderX 控制台不断打印错误：

```
10:35:47.096 读取失败:, {"errMsg":"串口未打开，请先调用 openSerial..."} at serial-test.vue:349
10:35:47.111 读取失败:, {"errMsg":"串口未打开，请先调用 openSerial..."} at serial-test.vue:349
10:35:47.142 读取失败:, {"errMsg":"串口未打开，请先调用 openSerial..."} at serial-test.vue:349
... (每 50ms 一条，无限循环)
```

### 问题分析

v1 页面直接调用 UTS 插件，页面自己管理轮询定时器：

```javascript
// v1 的实现方式（有问题）
const startReading = () => {
  readTimer = setInterval(() => {  // 页面自己创建定时器
    readSerial({ portId, ... })
  }, 50)
}

onBeforeUnmount(() => {
  stopStressTest()
  closeSerial({ portId })
  // 忘记调用 stopReading()！定时器还在后台运行！
})
```

当页面销毁时，`setInterval` 创建的定时器**并不会自动停止**，它仍在后台运行，不断调用 `readSerial`，而此时串口已关闭 → 报错。

### 根本原因

**手动管理定时器容易遗忘**。每个页面都要：
1. 创建定时器
2. 在 `onBeforeUnmount` 中清理定时器
3. 还要处理各种异常情况...

一旦漏掉任何一个地方，就会出 Bug。

### v1 vs v2 架构对比

| 对比项 | v1（直接调 UTS） | v2（通过 JS 服务层） |
|--------|-----------------|---------------------|
| **定时器管理** | 每个页面自己管理 | serialService 统一管理 |
| **页面销毁时** | 必须手动 stopReading() | 只需 off() 取消订阅 |
| **遗忘风险** | 高，容易忘记清理 | 低，服务层内部处理 |
| **定时器生命周期** | 跟随页面 | 跟随应用 |
| **代码复杂度** | 每页都写一遍 | 写一次，到处复用 |

### v1 的修复

```javascript
onBeforeUnmount(() => {
  stopReading()       // 必须加上这行！
  stopStressTest()
  closeSerial({ portId })
})
```

### v2 的设计

v2 使用 serialService.js 架构，页面不需要管理定时器：

```javascript
// v2 的实现方式（推荐）
import * as serial from '@/utils/serialService'

onMounted(() => {
  serial.on('frame', handleFrame)  // 订阅事件
})

onBeforeUnmount(() => {
  serial.off('frame', handleFrame) // 取消订阅即可
  // 不需要管定时器，serialService 内部处理
})
```

### 结论

这个 Bug 恰恰证明了 **JS 服务层架构的必要性**：

1. **职责分离**：定时器逻辑封装在服务层，页面专注 UI
2. **单一管理点**：定时器只在 serialService.js 里创建/销毁
3. **降低心智负担**：页面开发者不需要记住"离开页面要清理定时器"
4. **复用性**：多个页面共享同一个服务，不用重复写轮询代码

**v1 是临时方案，v2 才是正道！**

## 工作状态控制架构（v3）

> 更新日期：2026年2月12日

### 问题背景

在实现 v2 架构（serialService.js 服务层）后，发现一个新问题：

**现象**：App 启动后，控制台间歇性报错：

```
[JS Framework] Failed to execute the callback function:
Error: 返回值为：""; 请求参数为：{..."name":"readSerialByJs"...}
```

### 问题分析

v2 架构的设计是：**串口连接成功后立即启动轮询**。

```javascript
// v2 的问题代码
export function connect(options) {
  openSerial({
    success: (res) => {
      startPolling()  // ← 连接成功就开始轮询，永不停止
    }
  })
}
```

这带来几个问题：

1. **空转浪费**：在首页、课程列表等页面，轮询毫无意义
2. **回调失效**：页面切换、热更新时，回调 ID 可能失效，导致 `Failed to execute the callback function`
3. **语义不清**：用户还没开始训练，为什么要读数据？

### 根本原因：对下位机通信模型的误解

最初以为下位机是**主动上报**模式：
```
设备工作 → 持续发送数据 → App 轮询读取
```

实际上下位机是**请求-响应**模式：
```
App 发送命令 → 设备回复一帧 → 完成
不发命令 → 没有响应
```

**所以空转轮询不仅浪费资源，还永远读不到数据！**

### v3 架构：读发分离，按需启停

#### 核心思想

```
┌─────────────────────────────────────────┐
│  发送定时器（200ms 周期，5Hz）            │
│  send() → send() → send() → ...         │
│  不等待，不阻塞                           │
└─────────────────────────────────────────┘
                    ↕ 并行运行
┌─────────────────────────────────────────┐
│  读取定时器（50ms 周期）                  │
│  read() → read() → read() → ...         │
│  读到就处理，读空就跳过                   │
└─────────────────────────────────────────┘
```

- **读发分离**：发送只管发，不阻塞等待响应；读取只管读，读到就处理
- **并行运行**：两个定时器独立运行，不互相依赖
- **丢帧容忍**：1000 次采样丢 1-2 帧不影响曲线绘制，人眼无法察觉

#### 状态机

```
             ┌──────────────┐
             │   空闲状态    │  串口已连接，不轮询，不发送
             │   (IDLE)     │  
             └──────┬───────┘
                    │ 用户开启力量开关
                    ▼
             ┌──────────────┐
             │   工作状态    │  周期发送（5Hz）+ 轮询读取（50ms）
             │  (WORKING)   │  
             └──────┬───────┘
                    │ 用户关闭力量开关 / 离开页面
                    ▼
             ┌──────────────┐
             │   空闲状态    │
             └──────────────┘
```

#### 生命周期控制

| 时机 | 操作 | API |
|------|------|-----|
| App 启动 | 只连接串口，不启动任何定时器 | `connect()` |
| 开启力量开关 | 启动发送定时器 + 读取定时器 | `startWorking(force, mode)` |
| 调整力量滑块 | 更新下次发送的力量值（不重启定时器） | `updateWorkingForce(force)` |
| 关闭力量开关 | 停止工作状态，并启动安全停力序列 | `stopForce()` |
| 离开训练页面 | 同上，确保安全关闭 | `stopForce()` |
| App 退出 | 清理所有资源 | `cleanup()` |

### v2 vs v3 架构对比

| 对比项 | v2（连接即轮询） | v3（按需启停） |
|--------|-----------------|---------------|
| **轮询时机** | 连接成功就开始 | 用户开启力量开关才开始 |
| **发送时机** | 事件驱动（用户操作时） | 周期发送（5Hz） |
| **读发关系** | 分离 | 分离 + 并行 |
| **资源消耗** | 全程轮询，浪费资源 | 按需启停，节省资源 |
| **回调安全** | 页面切换可能失效 | 页面卸载时主动停止 |
| **实时曲线** | 不支持（只读不发） | 支持（周期发送获取采样点） |

### 为什么选择"读发分离"而不是"发送-等待"？

**发送-等待模式**（每次发送都阻塞等待响应）：
```javascript
const response = await sendAndReceive(command)  // 阻塞
```

**读发分离模式**（发送和读取独立运行）：
```javascript
setInterval(() => send(command), 200)  // 发送不等待
setInterval(() => read(), 50)          // 读取独立运行
```

| 对比项 | 发送-等待 | 读发分离 |
|--------|----------|----------|
| 实现复杂度 | 复杂（需要 async/await 同步） | 简单（两个独立定时器） |
| 丢帧影响 | 阻塞整个流程 | 无影响，继续运行 |
| 灵活性 | 发送频率=响应频率 | 可独立调整 |
| 性能 | 串行执行 | 并行执行 |

**结论**：读发分离更简单、更健壮、更灵活。

### 安全停力序列：为什么关闭动作收敛为 `stopForce()`

早期页面层的关闭逻辑并不统一，有几种常见写法：

- `stopWorking()`
- `stopWorking() + sendOnce(5, FORCE_MODE.OFF)`
- 页面自己再开一个 `setInterval` 连续补发 `OFF`

这些写法最大的问题不是"能不能发出一条 OFF"，而是**每个页面都在自己决定关闭语义**。最后导致：

1. 有的页面只是停掉工作定时器，但没有真正发关闭帧
2. 有的页面只补发 1 次 OFF，抗丢包能力弱
3. 有的页面自己维护 OFF 定时器，生命周期容易和页面逻辑打架
4. 关闭路径分散后，排查"为什么设备还在出力"会非常费劲

因此，当前的设计方向是：**把"停力"定义成一个服务层能力，而不是页面各写各的组合拳。**

`stopForce()` 的语义是：

1. 取消上一轮尚未结束的停力序列
2. 停止工作状态的周期发送
3. 立刻发送 1 次 `force=5, mode=OFF`
4. 按固定间隔继续补发 `OFF`
5. 到达指定时长后自动结束

默认参数：

- `duration = 2000ms`
- `interval = 200ms`

因此默认节奏可以理解成：

- 第 0ms：立即发送 1 次 OFF
- 第 200ms、400ms、600ms ...：继续补发 OFF
- 持续约 2 秒

从代码层面看，默认总发送次数大约是 **10 到 11 次**。之所以不是绝对固定值，是因为 `setInterval` 和 `setTimeout` 在 2000ms 边界时谁先进入调度，存在实现时序差异。

之所以必须连续补发，而不是只发 1 次，原因很现实：

1. 串口链路不是事务系统，不存在"对方一定收到并执行"的强保证
2. 页面切换、用户快速点击、底层插件调度，都会让单次 OFF 的可靠性下降
3. 对于停力来说，"宁可在短时间内多确认几次关闭"比"只发一枪然后赌它收到"更安全

这里还有一个经常容易看错的点：

- `stopWorking()` 的职责是"停止工作态周期发送"
- `stopForce()` 的职责是"执行完整停力动作"

也就是说，`stopWorking()` 不是关闭命令，它只是把"持续开力"停掉。真正负责让设备进入安全关闭状态的，是 `stopForce()`。

### `_cancelShutdownSequence()` 的定位

`_cancelShutdownSequence()` 不是发送函数，它只负责**终止正在进行中的停力连续发送**。

它会做三件事：

1. `clearInterval(shutdownTimer)`，停止后续 OFF 补发
2. `clearTimeout(shutdownTimeout)`，取消这轮停力序列的自动收尾
3. `shutdownEpoch = 0`，把这轮停力序列的身份标记清空

它主要在两个地方被调用：

1. `stopForce()` 开始前：避免两个停力序列重叠
2. `startWorking()` 开始前：如果用户在停力序列尚未结束时重新开机，先把旧的停力序列停掉

这个函数的作用可以概括为：

**把"还在持续广播 OFF"的那一整套动作中止，而不是再发送一条什么新命令。**

### 一次失败尝试：为什么 JS 串行发送队列不适合这个项目

在排查"设备像是同时收到开和关"时，曾尝试过一个看起来很合理的方案：

- 把所有 `writeSerial` 调用统一收口
- 在 JS 层做一个 FIFO 串行发送管道
- 前一个写完，再发下一个

这个方案在普通业务系统里通常是加分项，因为它解决了：

1. 多个调用点同时写串口，顺序不可控
2. 很难判断到底是谁先把命令发出去
3. 从"软件事务感"上看，统一排队比四处直写更整齐

但这套思路在本项目的"实时电机控制"场景里，实际效果是负面的。

#### 表面收益

串行发送队列最初想解决的是：

1. `startWorking()` 的周期发送
2. `stopForce()` 的 OFF 连发
3. `sendOnce()` 的单次命令

这些入口都可能在较短时间内交错触发。把它们先放进一个队列，直觉上会觉得：

- 至少不会"并发写"
- 顺序会更清楚
- 也更适合做日志追踪

这部分想法本身没有错。

#### 真正的问题：它改变了控制命令的时间语义

电机控制和普通消息队列最大的不同在于：

- 普通消息系统重视"每一条都按顺序送达"
- 实时控制系统重视"当前最新意图尽快生效"

而 FIFO 队列天然偏向第一种思路。

原来的发送节奏大致是：

- `startWorking()` 每 200ms 尽快发 1 次工作帧
- `stopForce()` 每 200ms 尽快发 1 次 OFF 帧
- 哪个动作先发生，哪个命令就尽快进入底层串口插件

虽然这套方式不够优雅，但它有一个关键优点：

**新状态的命令，能够比较快地把旧状态覆盖掉。**

一旦引入 JS 层串行队列，系统行为就变了：

1. 定时器仍然每 200ms 产生新命令
2. 但命令不再立即下发，而是先排队
3. 后面的命令必须等待前面的写入完成
4. 实际到达下位机的时间，不再等于"定时器触发时间"

这会直接把原本比较稳定的控制节奏，变成：

- `200ms`
- `200ms + 前一个 writeSerial 的完成时间`
- `200ms + 队列等待时间`
- 甚至再叠加其它命令插队造成的抖动

最终结果是，理论上的"每 200ms 一帧"会变成一种**带抖动的非稳定节奏**。

#### 为什么抖动会让电机表现成"开和关在对抗"

下位机看到的不是你的 JS 设计多优雅，它只看到一帧一帧真正到达的控制命令。

当节奏被队列拉乱后，设备可能感知到的是：

1. 一帧开力
2. 隔了一段不稳定时间后，一帧关力
3. 又隔了一段时间，一帧开力
4. 下一帧再被旧命令拖住

从软件日志看，可能还是"顺序正确"的；但从机械表现看，就会像：

- 输出不顺
- 发僵
- 顶住
- 甚至像两股相反的控制在同时拉扯

用户在真机上观察到的"不是安静地不动，而是明显像两方对抗的不动"，本质上就很符合这种**控制时序被破坏**后的表现。

#### 为什么串行队列删掉后反而恢复正常

删掉 JS 串行发送队列之后，系统回到了更接近原始设计的状态：

1. 定时器触发后，命令尽快直接交给底层串口层
2. 不再引入额外的 JS 队列等待
3. 新的状态切换更快传达到设备

这并不意味着"队列思想一定错误"，而是说明：

**这个项目的控制命令，不适合用 FIFO 去保证"历史命令都认真发完"。**

这里真正需要的是：

- 保证最新状态尽快生效
- 允许旧状态命令失效
- 不让过期命令阻塞当前状态

也就是说，本项目更适合的是"状态覆盖模型"，而不是"消息排队模型"。

#### 这个失败尝试留下来的结论

对本项目来说，JS 层串行发送队列的问题不是代码写得不好，而是设计目标和硬件控制目标不一致：

1. 它强调顺序送达
2. 但我们真正需要的是最新状态优先
3. 它引入了额外延迟和抖动
4. 这些抖动足以改变电机行为

因此，最终决策是：

- **移除 JS 层 FIFO 串行发送管道**
- **保留"旧命令失效"能力**
- 让发送重新回到更接近实时的节奏

### 命令世代（`commandEpoch`）设计：为什么保留，解决了什么

在移除 JS 串行发送队列后，仍然保留了一个很重要的设计：**命令世代**。

这个设计的目标不是让所有命令排队，而是给每一轮"当前有效控制状态"发一个编号。只要状态切换了，旧编号对应的命令就自动作废。

#### 核心思路

服务层维护一个全局递增值：

```javascript
let commandEpoch = 0

function nextCommandEpoch() {
  commandEpoch += 1
  return commandEpoch
}
```

然后在不同发送场景下，为当前状态记录自己的 epoch：

- `workingEpoch`：当前工作状态的发送世代
- `shutdownEpoch`：当前停力序列的发送世代

每次真正调用 `writeSerial` 之前，都会做一次检查：

```javascript
if (epoch !== undefined && epoch !== null && epoch !== commandEpoch) {
  resolve({ skipped: true, stale: true })
  return
}
```

这句检查的含义是：

- 如果这条命令属于旧世代
- 那就不要再发了
- 直接把它当成"过期命令"跳过

#### 它在几个关键入口里的作用

`startWorking()` 会：

1. 先取消正在进行中的停力序列
2. `stopWorking()` 停止旧的工作发送
3. 调用 `nextCommandEpoch()`，生成新的 `workingEpoch`
4. 后续工作帧都带着这个 `workingEpoch` 发出去

`stopWorking()` 会：

1. 调用 `nextCommandEpoch()`
2. 这一步会让之前的工作帧全部失效
3. 再清掉工作定时器

`stopForce()` 会：

1. 取消旧的停力序列
2. `stopWorking()` 先让旧的工作状态失效
3. 再调用 `nextCommandEpoch()` 生成新的 `shutdownEpoch`
4. 后续这一轮 OFF 补发都带着 `shutdownEpoch`

因此，"工作状态"和"停力状态"并不是靠一个布尔值互相覆盖，而是靠不同世代去标识：

- 当前到底哪一轮命令还有效
- 哪些命令已经是旧状态残留

#### 典型场景：关机 2 秒内又重新开机

这是命令世代设计最有价值的场景。

假设时间线如下：

1. 用户点"关"
2. `stopForce()` 启动，开始每 200ms 补发 OFF
3. 还没到 2 秒，用户又点"开"
4. `startWorking()` 被调用

如果没有命令世代，风险是：

- 虽然 `shutdownTimer` 被清掉了
- 但某些旧 OFF 命令可能仍然会继续向下发送
- 新的开力命令和旧的 OFF 会继续打架

有了命令世代之后，逻辑就变成：

1. 第一次关机序列使用 `shutdownEpoch = N`
2. 重新开机时，`startWorking()` 把全局 `commandEpoch` 推进到 `N + 1`
3. 此时旧的 `shutdownEpoch = N` 已经过期
4. 即使还有没来得及真正下发的旧 OFF 命令，它们在发送前也会被判定为 `stale`
5. 这些过期 OFF 会被跳过，不再写入串口

这就是命令世代真正解决的问题：

**不是撤销已经发出的命令，而是阻止"还没发出去但已经过期的命令"继续下发。**

#### 它和串行队列的本质区别

这两者很容易被误认为是一套东西，其实不是。

串行队列做的是：

- 让所有命令先排队
- 前一个完成后，后一个再发

命令世代做的是：

- 不改变发送链路的实时性
- 只在真正下发前做一次"这条命令还算数吗"的判断

所以命令世代的副作用比串行队列小得多：

1. 不额外引入 FIFO 等待
2. 不改变 200ms 周期本身的时间语义
3. 只负责让旧状态命令失效

这也是为什么最终选择：

- 删掉 JS 串行发送队列
- 保留 `commandEpoch`

#### 它的边界：能做什么，不能做什么

命令世代并不是万能的，必须清楚它的边界。

它能做的是：

1. 让旧状态但尚未下发的命令失效
2. 避免停力序列和开力序列在 JS 层继续交错下发
3. 让服务层对"当前有效控制状态"有明确定义

它做不到的是：

1. 撤回已经调用过 `writeSerial` 的命令
2. 撤回已经进入原生插件缓冲区的命令
3. 撤回已经到达下位机并被执行的命令

所以当用户快速执行 `关 -> 开 -> 关` 时，要这样理解：

- 第一轮关机已经真正发出去的那几帧 OFF，无法追回
- 但第一轮关机里"后续还没发出去的 OFF"可以被打断
- 第二轮开机和第二轮关机，会从新的世代继续运行

这就是为什么命令世代是一个**必要但有限**的保护机制。它能减少旧命令拖尾，但不能把硬件世界里已经发生的发送历史抹掉。

### 当前设计结论

基于真实设备调试的结果，当前服务层在"开力/停力控制"上的设计原则应当明确写下来：

1. **关闭动作统一走 `stopForce()`，不要让页面各自拼装关闭逻辑**
2. **不要在 JS 层引入 FIFO 串行发送队列**
3. **保留命令世代（`commandEpoch`）作为旧命令失效机制**
4. **优先保证最新控制状态尽快生效，而不是保证所有历史命令都发完**
5. **对实时电机控制来说，发送节奏稳定性比"消息队列式的顺序感"更重要**

## ES Module 单例机制

### 为什么多个页面共享同一个 serialService？

在 uni-app（以及所有现代 JS 框架）中，ES Module 规范保证每个模块**只执行一次**，后续 `import` 都返回同一个模块对象的引用，而非拷贝。

```javascript
// serialService.js
let portId = 0           // 模块级变量
let isConnected = false  // 只存在一份

export function connect() { ... }
export function send() { ... }
```

```javascript
// 页面A
import serialService from '@/utils/serialService'
serialService.connect()  // portId 变成 1

// 页面B
import serialService from '@/utils/serialService'
console.log(serialService.getStatus().portId)  // 输出 1，不是 0
```

### 内存模型

```
+--------------------------------------------------+
|                    内存                           |
|                                                  |
|   serialService.js (单例)                         |
|   +------------------------------+               |
|   | portId = 1                   |               |
|   | isConnected = true           |               |
|   | listeners = { frame: [...] } |               |
|   | rxBuffer = "73..."           |               |
|   +------------------------------+               |
|         ^           ^           ^                |
|         |           |           |                |
|    +----+---+  +----+---+  +----+---+            |
|    | 页面 A |  | 页面 B |  | 页面 C |            |
|    | import |  | import |  | import |            |
|    +--------+  +--------+  +--------+            |
|    (引用)      (引用)       (引用)               |
+--------------------------------------------------+
```

### 单例设计要点

| 设计点 | 实现方式 |
|--------|----------|
| 单例连接 | 模块级 `portId`, `isConnected` |
| 事件广播 | `listeners` 数组，多页面订阅 |

| 生命周期 | `cleanup()` 在 App 退出时调用 |

### 页面使用示例

> ❗ 注意：串口连接在 **App.vue 的 onLaunch** 中完成，页面只需订阅事件 + 启动工作状态。

```javascript
// freeTraining.vue 或其他业务页面
import { on, off, startWorking, stopWorking, getStatus } from '@/utils/serialService'

onMounted(() => {
  // 订阅帧事件
  on('frame', handleFrame)
  
  // 检查连接状态（App.vue 已连接）
  const status = getStatus()
  if (status.isConnected) {
    // 启动工作状态：周期发送 + 轮询读取
    startWorking(0, 1, 200)
  } else {
    console.warn('串口未连接，请检查 App.vue 初始化')
  }
})

onBeforeUnmount(() => {
  // 取消订阅，但不断开连接（其他页面可能还在用）
  off('frame', handleFrame)
  stopWorking()
})
```

**连接职责分离：**

| 层级 | 职责 | 类比 C++ |
|------|------|----------|
| **App.vue** | 连接串口（全局单例） | 构造函数 |
| **页面** | 订阅事件 + 启动工作状态 | 成员方法 |

**结论**：uni-app 中 `import` 模块天然就是单例，serialService.js 的设计充分利用了这一特性，无需额外的单例模式代码。

### 协议帧类型

本系统支持两种帧格式：

| 帧类型 | 包类型字节 | 方向 | 长度 | 用途 |
|--------|-----------|------|------|------|
| **A9 帧** | `0x09` | 设备 → App | 96 字节 | 上报设备状态、传感器数据 |
| **D180 帧** | `0xB4` | App → 设备 | 9 字节 | 下发控制指令 |

### CRC 校验算法

> ❗ 注意：本项目使用的是**特殊口径的 CRC-32 算法**，不是简单异或！

**算法参数：**
- 多项式: `0x04C11DB7`
- 初始值: `0xFFFFFFFF`
- 结果异或: `0x00000000`
- 反射: 无 (MSB-first)
- 取值: **低 16 位**，大端存储

**预处理步骤：**
1. 取 CRC 字段之前的所有字节
2. 补 0 到 4 的倍数
3. 每 4 字节分组倒序
4. 计算 CRC-32
5. 取低 16 位

```javascript
// 预处理：补 0 + 每 4 字节倒序
function preprocessPadZeroReverse4(src) {
  const padded = (src.length + 3) & ~3  // 向上取整到 4 的倍数
  const out = new Uint8Array(padded)
  out.set(src)
  
  for (let i = 0; i < padded; i += 4) {
    const t0 = out[i], t1 = out[i + 1]
    out[i] = out[i + 3]
    out[i + 1] = out[i + 2]
    out[i + 2] = t1
    out[i + 3] = t0
  }
  return out
}

// CRC-32 计算 (MSB-first, 非反射)
function crc32NonRefNoXor(data) {
  let crc = 0xFFFFFFFF
  for (let i = 0; i < data.length; i++) {
    crc ^= (data[i] << 24)
    for (let j = 0; j < 8; j++) {
      if ((crc & 0x80000000) !== 0) {
        crc = ((crc << 1) ^ 0x04C11DB7) >>> 0
      } else {
        crc = (crc << 1) >>> 0
      }
    }
  }
  return crc >>> 0
}

// 计算 CRC-16
function computeCrc16Fixed(pkt, crcOffset) {
  const data = pkt.slice(0, crcOffset)
  const pre = preprocessPadZeroReverse4(data)
  const crc32 = crc32NonRefNoXor(pre)
  return crc32 & 0xFFFF  // 取低 16 位
}
```

## 完整使用示例

```vue
<template>
  <view class="container">
    <button @click="handleListDevices">列举设备</button>
    <button @click="handleOpen">打开串口</button>
    <button @click="handleWrite">写入数据</button>
    <button @click="handleRead">读取数据</button>
    <button @click="handleClose">关闭串口</button>
    <text>{{ statusMsg }}</text>
  </view>
</template>

<script setup lang="ts">
import { ref } from 'vue'
import { 
  openSerial, 
  writeSerial, 
  readSerial, 
  closeSerial,
  listDevices
} from '@/uni_modules/wzl-serialbridge'

const statusMsg = ref<string>('')
let currentPortId = 0

// 列举设备
const handleListDevices = () => {
  listDevices({
    success: (res) => {
      statusMsg.value = `找到设备: ${res.devices.join(', ')}`
    }
  })
}

// 打开串口
const handleOpen = () => {
  openSerial({
    path: "/dev/ttyS0",
    config: {
      baudRate: 115200,
      dataBits: 8,
      stopBits: 1,
      parity: 'none'
    },
    success: (res) => {
      currentPortId = res.portId
      statusMsg.value = `串口已打开,ID: ${currentPortId}`
    },
    fail: (err) => {
      statusMsg.value = `打开失败: ${err.errMsg}`
    }
  })
}

// 写入数据
const handleWrite = () => {
  if (currentPortId === 0) {
    statusMsg.value = "请先打开串口"
    return
  }
  
  writeSerial({
    portId: currentPortId,
    data: "01 03 00 00 00 01 84 0A",
    format: 'hex',
    success: (res) => {
      statusMsg.value = `写入成功: ${res.bytesWritten} 字节`
    },
    fail: (err) => {
      statusMsg.value = `写入失败: ${err.errMsg}`
    }
  })
}

// 读取数据
const handleRead = () => {
  if (currentPortId === 0) {
    statusMsg.value = "请先打开串口"
    return
  }
  
  readSerial({
    portId: currentPortId,
    length: 128,
    format: 'hex',
    timeout: 2000,
    success: (res) => {
      statusMsg.value = `读取到: ${res.data} (${res.bytesRead}字节)`
    },
    fail: (err) => {
      statusMsg.value = `读取失败: ${err.errMsg}`
    }
  })
}

// 关闭串口
const handleClose = () => {
  if (currentPortId === 0) {
    statusMsg.value = "串口未打开"
    return
  }
  
  closeSerial({
    portId: currentPortId,
    success: () => {
      statusMsg.value = "串口已关闭"
      currentPortId = 0
    },
    fail: (err) => {
      statusMsg.value = `关闭失败: ${err.errMsg}`
    }
  })
}
</script>
```

## 注意事项

1. **权限配置**: 需要在 `manifest.json` 中配置存储权限:
   ```json
   "android": {
     "permissions": [
       "<uses-permission android:name=\"android.permission.READ_EXTERNAL_STORAGE\"/>",
       "<uses-permission android:name=\"android.permission.WRITE_EXTERNAL_STORAGE\"/>"
     ]
   }
   ```

2. **自定义基座**: 由于使用了 AAR 库,需要制作自定义基座才能运行。

3. **设备路径**: Android 常见串口设备路径:
   - `/dev/ttyS0`, `/dev/ttyS1` - 系统串口
   - `/dev/ttyUSB0` - USB 转串口
   - `/dev/ttyAMA0` - 树莓派等设备

4. **数据格式**: 
   - HEX 格式支持空格分隔,如 `"01 03 00 00"`
   - Base64 格式需要标准 Base64 编码
