# wzl-serialbridge 串口通信插件

基于 UTS 的 Android 串口通信插件,支持 UART 设备的读写操作。

## 为什么需要 UTS 插件？

### 前端语言无法直接访问串口

在 uni-app 中，JavaScript/Vue 运行在受限的沙箱环境里，存在以下限制：

| 限制 | 说明 |
|------|------|
| 浏览器沙箱 | JS 运行在沙箱里，不能直接访问硬件 |
| 没有串口 API | uni-app 的 JS 引擎没有提供访问 /dev/ttyS3 的能力 |
| 权限问题 | 串口是 Linux 设备文件，需要系统级权限 |

### 调用链路

```
JS (前端)
   |
   | uni-app 插件机制
   v
UTS (桥接层，编译成 Kotlin)
   |
   | import AAR
   v
Kotlin AAR (原生层)
   |
   | Java FileInputStream / NDK
   v
Linux 内核  -->  /dev/ttyS3 (串口硬件)
```

## UTS 插件与 AAR 的关系

### 什么是 UTS？

UTS (Uni TypeScript) 是 uni-app 提供的跨平台原生插件开发语言，语法类似 TypeScript，但可以直接编译成各平台的原生代码：

| 平台 | UTS 编译目标 |
|------|-------------|
| Android | Kotlin |
| iOS | Swift |
| 鸿蒙 | ArkTS |

### 什么是 AAR？

AAR (Android Archive) 是打包好的 Android 原生库，包含 Kotlin/Java 编译后的字节码。

### 本插件的架构

```
serialbridge.aar (Kotlin)           index.uts (UTS)                  serial-test.vue (JS)
+--------------------+              +------------------+             +------------------+
| class SerialPort { |  import      | import SerialPort|   export    | import {         |
|   open()           | <----------  | from 'com...aar' | <---------- |   openSerial     |
|   read()           |              |                  |             | } from '...'     |
|   write()          |              | fun openSerial() |             |                  |
|   close()          |              |   SerialPort()   |             | openSerial({...})|
| }                  |              | }                |             |                  |
+--------------------+              +------------------+             +------------------+
     纯 Android 代码                   桥接/适配层                       业务代码
```

### 为什么需要 AAR？

UTS 虽然能写 Kotlin，但有些场景需要 AAR：

| 场景 | 说明 |
|------|------|
| 复用现有库 | 已有成熟的 Kotlin/Java 串口库，直接打包成 AAR 引入 |
| 复杂原生功能 | JNI、NDK、硬件驱动等，UTS 不方便写 |
| 保护代码 | AAR 是编译后的二进制，比 UTS 源码更难反编译 |
| 团队分工 | Android 开发写 AAR，前端开发写 UTS 对接 |

### 简单总结

- UTS = uni-app 的"翻译官"，让 JS 能调用原生代码
- AAR = 打包好的 Android 原生库（Kotlin/Java 编译产物）
- 关系：UTS 通过 import 语句导入 AAR 中的类，然后封装成 uni-app 风格的 API 供 JS 调用

## 推荐的分层架构设计

### 背景与问题

本项目在开发初期使用 Android 原生开发（Android Studio），当时的通信架构是：
- 串口通信对象放在长生命周期的组件（Service）里统一维护
- Activity 之间不直接"传串口数据"
- 把 Service 当成"通信中枢"：各个 Activity 绑定到同一个 Service

现在迁移到 uni-app Vue 框架后，需要重新设计通信架构。

### 当前临时方案的问题

当前 serial-test.vue 直接调用 UTS 插件：

```
serial-test.vue  ──直接调用──>  UTS 插件 (wzl-serialbridge)
```

这种方式存在问题：
- 每个页面都要自己管理 portId
- 每个页面都要自己写轮询逻辑
- 页面切换后串口状态丢失
- 帧解析逻辑（拆包/粘包）要到处复制

### 推荐的分层架构

```
+---------------------------------------------------------------+
|  Vue 页面层 (free-training.vue, serial-test.vue ...)           |
|  - UI 交互                                                     |
|  - 调用 serialService                                          |
+---------------------------------------------------------------+
                              |
                              v
+----------------------------------------------------------------+
|  JS 服务层 (serialService.js) <-- 新增的，纯 JS 文件            |
|  - 单例管理                                                    |
|  - 事件订阅/分发                                               |
|  - 帧解析(拆包/粘包)                                           |
|  - 调用 UTS 插件                                               |
+---------------------------------------------------------------+
                              |
                              v
+----------------------------------------------------------------+
|  UTS 插件层 (wzl-serialbridge) <-- 现有的，不用改！             |
|  - openSerial / closeSerial                                   |
|  - readSerial / writeSerial                                   |
|  - 调用 AAR                                                   |
+----------------------------------------------------------------+
                              |
                              v
+----------------------------------------------------------------+
|  AAR 层 (serialbridge.aar) <-- 现有的，不用改！                 |
|  - Kotlin 代码                                                 |
|  - 直接操作 /dev/ttyS*                                         |
+----------------------------------------------------------------+
```

### 各层职责划分

| 层 | 文件 | 职责 | 需要修改？ |
|---|------|------|----------|
| AAR | serialbridge.aar | 底层串口操作（打开、读、写、关闭） | 不用改 |
| UTS | wzl-serialbridge/*.uts | 把 AAR 暴露给 JS | 不用改 |
| JS 服务 | serialService.js | 单例+事件+帧解析 | 新增 |
| Vue 页面 | *.vue | UI 交互 | 改调用方式 |

### 功能应该放在哪一层？

| 功能 | 放哪里 | 原因 |
|------|--------|------|
| 打开/关闭串口 | AAR + UTS | 已有，不动 |
| 读/写字节 | AAR + UTS | 已有，不动 |
| HEX/Base64 转换 | UTS | 已有，不动 |
| 连接单例管理 | JS 服务层 | Vue 框架里更方便 |
| 帧解析(73...65) | JS 服务层 | 业务逻辑，纯 JS 就能做 |
| 事件订阅/分发 | JS 服务层 | 多页面共享数据 |
| 轮询定时器 | JS 服务层 | 集中管理 |
| UI 显示 | Vue 页面 | 展示层 |

### 关键结论

**不需要改 AAR，不需要改 UTS 插件！**

只需要在 JS 层加一个服务模块（serialService.js），它调用现有的 UTS 插件，然后提供更高级的封装：

```javascript
// serialService.js 核心思路

import { 
  openSerial,   // <-- 调用现有的 UTS 插件
  writeSerial, 
  readSerial, 
  closeSerial 
} from '@/uni_modules/wzl-serialbridge'

// 然后封装成更高级的 API
export function connect(path) {
  openSerial({ path, ... })  // 调用 UTS
}

export function send(data) {
  writeSerial({ ... })       // 调用 UTS
}

export function on(event, callback) {
  // 事件订阅机制
}
```

### 加了 JS 服务层之后的调用关系

```
serial-test.vue   ──调用──>  serialService.js  ──调用──>  UTS 插件
free-training.vue ─────────┘
其他页面 ─────────────────┘
```

好处：
- 串口连接全局唯一（单例）
- 轮询、帧解析只写一次
- 页面只需要订阅事件
- 页面切换不影响串口连接
- 关系：UTS 通过 import 语句导入 AAR 中的类，然后封装成 uni-app 风格的 API 供 JS 调用

## 实践案例：定时器泄漏问题

### 问题背景

2026年2月9日，在开发串口测试页面时遇到一个典型 Bug：

**现象**：从 serial-test.vue（v1）页面跳转到其他页面后，HBuilderX 控制台不断打印错误：

```
10:35:47.096 读取失败:, {"errMsg":"串口未打开，请先调用 openSerial..."} at serial-test.vue:349
10:35:47.111 读取失败:, {"errMsg":"串口未打开，请先调用 openSerial..."} at serial-test.vue:349
10:35:47.142 读取失败:, {"errMsg":"串口未打开，请先调用 openSerial..."} at serial-test.vue:349
... (每 50ms 一条，无限循环)
```

### 问题分析

v1 页面直接调用 UTS 插件，页面自己管理轮询定时器：

```javascript
// v1 的实现方式（有问题）
const startReading = () => {
  readTimer = setInterval(() => {  // 页面自己创建定时器
    readSerial({ portId, ... })
  }, 50)
}

onBeforeUnmount(() => {
  stopStressTest()
  closeSerial({ portId })
  // ❌ 忘记调用 stopReading()！定时器还在后台运行！
})
```

当页面销毁时，`setInterval` 创建的定时器**并不会自动停止**，它仍在后台运行，不断调用 `readSerial`，而此时串口已关闭 → 报错。

### 根本原因

**手动管理定时器容易遗忘**。每个页面都要：
1. 创建定时器
2. 在 `onBeforeUnmount` 中清理定时器
3. 还要处理各种异常情况...

一旦漏掉任何一个地方，就会出 Bug。

### v1 vs v2 架构对比

| 对比项 | v1（直接调 UTS） | v2（通过 JS 服务层） |
|--------|-----------------|---------------------|
| **定时器管理** | 每个页面自己管理 | serialService 统一管理 |
| **页面销毁时** | 必须手动 stopReading() | 只需 off() 取消订阅 |
| **遗忘风险** | ⚠️ 高，容易忘记清理 | ✅ 低，服务层内部处理 |
| **定时器生命周期** | 跟随页面 | 跟随应用 |
| **代码复杂度** | 每页都写一遍 | 写一次，到处复用 |

### v1 的修复

```javascript
onBeforeUnmount(() => {
  stopReading()       // ✅ 必须加上这行！
  stopStressTest()
  closeSerial({ portId })
})
```

### v2 的设计

v2 使用 serialService.js 架构，页面不需要管理定时器：

```javascript
// v2 的实现方式（推荐）
import * as serial from '@/utils/serialService'

onMounted(() => {
  serial.on('frame', handleFrame)  // 订阅事件
})

onBeforeUnmount(() => {
  serial.off('frame', handleFrame) // 取消订阅即可
  // 不需要管定时器，serialService 内部处理
})
```

### 结论

这个 Bug 恰恰证明了 **JS 服务层架构的必要性**：

1. **职责分离**：定时器逻辑封装在服务层，页面专注 UI
2. **单一管理点**：定时器只在 serialService.js 里创建/销毁
3. **降低心智负担**：页面开发者不需要记住"离开页面要清理定时器"
4. **复用性**：多个页面共享同一个服务，不用重复写轮询代码

**v1 是临时方案，v2 才是正道！**

## API 文档

### 1. listDevices - 列举可用设备

```typescript
listDevices({
  prefixes: ["/dev/ttyS", "/dev/ttyUSB"], // 可选
  success: (res) => {
    console.log("可用设备:", res.devices)
    // 输出: ["/dev/ttyS0", "/dev/ttyS1", "/dev/ttyUSB0"]
  },
  fail: (err) => {
    console.error("失败:", err.errMsg)
  }
})
```

### 2. openSerial - 打开串口

```typescript
const options: OpenOptions = {
  path: "/dev/ttyS0",
  config: {
    baudRate: 115200,  // 波特率
    dataBits: 8,       // 数据位: 5|6|7|8
    stopBits: 1,       // 停止位: 1|2
    parity: 'none'     // 校验位: 'none'|'odd'|'even'
  },
  success: (res) => {
    const portId = res.portId
    console.log("打开成功,句柄ID:", portId)
    // 保存 portId 用于后续操作
  },
  fail: (err) => {
    console.error("打开失败:", err.errMsg)
  }
}

openSerial(options)
```

### 3. writeSerial - 写入数据

```typescript
// 写入 HEX 数据
const writeOptions: WriteOptions = {
  portId: 1,                    // openSerial 返回的句柄ID
  data: "01 03 00 00 00 01",   // HEX 字符串(自动去空格)
  format: 'hex',                // 'hex' | 'base64'
  timeout: 1000,                // 超时时间(毫秒)
  success: (res) => {
    console.log("写入成功:", res.bytesWritten, "字节")
  },
  fail: (err) => {
    console.error("写入失败:", err.errMsg)
  }
}

writeSerial(writeOptions)

// 或写入 Base64 数据
writeSerial({
  portId: 1,
  data: "SGVsbG8gV29ybGQ=",  // Base64 编码的数据
  format: 'base64',
  timeout: 1000,
  success: (res) => console.log("写入成功")
})
```

### 4. readSerial - 读取数据

```typescript
const readOptions: ReadOptions = {
  portId: 1,           // 句柄ID
  length: 1024,        // 最大读取字节数
  format: 'hex',       // 返回格式: 'hex' | 'base64'
  timeout: 1000,       // 超时时间
  success: (res) => {
    console.log("读取到数据:", res.data)
    console.log("字节数:", res.bytesRead)
    // HEX 格式输出: "010300000001"
    // Base64 格式输出: "AQMAAAAB"
  },
  fail: (err) => {
    console.error("读取失败:", err.errMsg)
  }
}

readSerial(readOptions)
```

### 5. closeSerial - 关闭串口

```typescript
closeSerial({
  portId: 1,
  success: (res) => {
    console.log("关闭成功")
  },
  fail: (err) => {
    console.error("关闭失败:", err.errMsg)
  }
})
```

## 完整使用示例

```vue
<template>
  <view class="container">
    <button @click="handleListDevices">列举设备</button>
    <button @click="handleOpen">打开串口</button>
    <button @click="handleWrite">写入数据</button>
    <button @click="handleRead">读取数据</button>
    <button @click="handleClose">关闭串口</button>
    <text>{{ statusMsg }}</text>
  </view>
</template>

<script setup lang="ts">
import { ref } from 'vue'
import { 
  openSerial, 
  writeSerial, 
  readSerial, 
  closeSerial,
  listDevices
} from '@/uni_modules/wzl-serialbridge'

const statusMsg = ref<string>('')
let currentPortId = 0

// 列举设备
const handleListDevices = () => {
  listDevices({
    success: (res) => {
      statusMsg.value = `找到设备: ${res.devices.join(', ')}`
    }
  })
}

// 打开串口
const handleOpen = () => {
  openSerial({
    path: "/dev/ttyS0",
    config: {
      baudRate: 115200,
      dataBits: 8,
      stopBits: 1,
      parity: 'none'
    },
    success: (res) => {
      currentPortId = res.portId
      statusMsg.value = `串口已打开,ID: ${currentPortId}`
    },
    fail: (err) => {
      statusMsg.value = `打开失败: ${err.errMsg}`
    }
  })
}

// 写入数据
const handleWrite = () => {
  if (currentPortId === 0) {
    statusMsg.value = "请先打开串口"
    return
  }
  
  writeSerial({
    portId: currentPortId,
    data: "01 03 00 00 00 01 84 0A",
    format: 'hex',
    success: (res) => {
      statusMsg.value = `写入成功: ${res.bytesWritten} 字节`
    },
    fail: (err) => {
      statusMsg.value = `写入失败: ${err.errMsg}`
    }
  })
}

// 读取数据
const handleRead = () => {
  if (currentPortId === 0) {
    statusMsg.value = "请先打开串口"
    return
  }
  
  readSerial({
    portId: currentPortId,
    length: 128,
    format: 'hex',
    timeout: 2000,
    success: (res) => {
      statusMsg.value = `读取到: ${res.data} (${res.bytesRead}字节)`
    },
    fail: (err) => {
      statusMsg.value = `读取失败: ${err.errMsg}`
    }
  })
}

// 关闭串口
const handleClose = () => {
  if (currentPortId === 0) {
    statusMsg.value = "串口未打开"
    return
  }
  
  closeSerial({
    portId: currentPortId,
    success: () => {
      statusMsg.value = "串口已关闭"
      currentPortId = 0
    },
    fail: (err) => {
      statusMsg.value = `关闭失败: ${err.errMsg}`
    }
  })
}
</script>
```

## 注意事项

1. **权限配置**: 需要在 `manifest.json` 中配置存储权限:
   ```json
   "android": {
     "permissions": [
       "<uses-permission android:name=\"android.permission.READ_EXTERNAL_STORAGE\"/>",
       "<uses-permission android:name=\"android.permission.WRITE_EXTERNAL_STORAGE\"/>"
     ]
   }
   ```

2. **自定义基座**: 由于使用了 AAR 库,需要制作自定义基座才能运行。

3. **设备路径**: Android 常见串口设备路径:
   - `/dev/ttyS0`, `/dev/ttyS1` - 系统串口
   - `/dev/ttyUSB0` - USB 转串口
   - `/dev/ttyAMA0` - 树莓派等设备

4. **数据格式**: 
   - HEX 格式支持空格分隔,如 `"01 03 00 00"`
   - Base64 格式需要标准 Base64 编码
