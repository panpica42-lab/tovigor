# wzl-serialbridge 串口通信插件

基于 UTS 的 Android 串口通信插件,支持 UART 设备的读写操作。

## 为什么需要 UTS 插件？

### 前端语言无法直接访问串口

在 uni-app 中，JavaScript/Vue 运行在受限的沙箱环境里，存在以下限制：

| 限制 | 说明 |
|------|------|
| 浏览器沙箱 | JS 运行在沙箱里，不能直接访问硬件 |
| 没有串口 API | uni-app 的 JS 引擎没有提供访问 /dev/ttyS3 的能力 |
| 权限问题 | 串口是 Linux 设备文件，需要系统级权限 |

### 调用链路

```
JS (前端)
   |
   | uni-app 插件机制
   v
UTS (桥接层，编译成 Kotlin)
   |
   | import AAR
   v
Kotlin AAR (原生层)
   |
   | Java FileInputStream / NDK
   v
Linux 内核  -->  /dev/ttyS3 (串口硬件)
```

## UTS 插件与 AAR 的关系

### 什么是 UTS？

UTS (Uni TypeScript) 是 uni-app 提供的跨平台原生插件开发语言，语法类似 TypeScript，但可以直接编译成各平台的原生代码：

| 平台 | UTS 编译目标 |
|------|-------------|
| Android | Kotlin |
| iOS | Swift |
| 鸿蒙 | ArkTS |

### 什么是 AAR？

AAR (Android Archive) 是打包好的 Android 原生库，包含 Kotlin/Java 编译后的字节码。

### 本插件的架构

```
serialbridge.aar (Kotlin)           index.uts (UTS)                  serial-test.vue (JS)
+--------------------+              +------------------+             +------------------+
| class SerialPort { |  import      | import SerialPort|   export    | import {         |
|   open()           | <----------  | from 'com...aar' | <---------- |   openSerial     |
|   read()           |              |                  |             | } from '...'     |
|   write()          |              | fun openSerial() |             |                  |
|   close()          |              |   SerialPort()   |             | openSerial({...})|
| }                  |              | }                |             |                  |
+--------------------+              +------------------+             +------------------+
     纯 Android 代码                   桥接/适配层                       业务代码
```

### 为什么需要 AAR？

UTS 虽然能写 Kotlin，但有些场景需要 AAR：

| 场景 | 说明 |
|------|------|
| 复用现有库 | 已有成熟的 Kotlin/Java 串口库，直接打包成 AAR 引入 |
| 复杂原生功能 | JNI、NDK、硬件驱动等，UTS 不方便写 |
| 保护代码 | AAR 是编译后的二进制，比 UTS 源码更难反编译 |
| 团队分工 | Android 开发写 AAR，前端开发写 UTS 对接 |

### 简单总结

- UTS = uni-app 的"翻译官"，让 JS 能调用原生代码
- AAR = 打包好的 Android 原生库（Kotlin/Java 编译产物）
- 关系：UTS 通过 import 语句导入 AAR 中的类，然后封装成 uni-app 风格的 API 供 JS 调用

## 推荐的分层架构设计

### 背景与问题

本项目在开发初期使用 Android 原生开发（Android Studio），当时的通信架构是：
- 串口通信对象放在长生命周期的组件（Service）里统一维护
- Activity 之间不直接"传串口数据"
- 把 Service 当成"通信中枢"：各个 Activity 绑定到同一个 Service

现在迁移到 uni-app Vue 框架后，需要重新设计通信架构。

### 当前临时方案的问题

当前 serial-test.vue 直接调用 UTS 插件：

```
serial-test.vue  ──直接调用──>  UTS 插件 (wzl-serialbridge)
```

这种方式存在问题：
- 每个页面都要自己管理 portId
- 每个页面都要自己写轮询逻辑
- 页面切换后串口状态丢失
- 帧解析逻辑（拆包/粘包）要到处复制

### 推荐的分层架构

```
+---------------------------------------------------------------+
|  Vue 页面层 (free-training.vue, serial-test.vue ...)           |
|  - UI 交互                                                     |
|  - 调用 serialService                                          |
+---------------------------------------------------------------+
                              |
                              v
+----------------------------------------------------------------+
|  JS 服务层 (serialService.js) <-- 新增的，纯 JS 文件            |
|  - 单例管理                                                    |
|  - 事件订阅/分发                                               |
|  - 帧解析(拆包/粘包)                                           |
|  - 调用 UTS 插件                                               |
+---------------------------------------------------------------+
                              |
                              v
+----------------------------------------------------------------+
|  UTS 插件层 (wzl-serialbridge) <-- 现有的，不用改！             |
|  - openSerial / closeSerial                                   |
|  - readSerial / writeSerial                                   |
|  - 调用 AAR                                                   |
+----------------------------------------------------------------+
                              |
                              v
+----------------------------------------------------------------+
|  AAR 层 (serialbridge.aar) <-- 现有的，不用改！                 |
|  - Kotlin 代码                                                 |
|  - 直接操作 /dev/ttyS*                                         |
+----------------------------------------------------------------+
```

### 各层职责划分

| 层 | 文件 | 职责 | 需要修改？ |
|---|------|------|----------|
| AAR | serialbridge.aar | 底层串口操作（打开、读、写、关闭） | 不用改 |
| UTS | wzl-serialbridge/*.uts | 把 AAR 暴露给 JS | 不用改 |
| JS 服务 | serialService.js | 单例+事件+帧解析 | 新增 |
| Vue 页面 | *.vue | UI 交互 | 改调用方式 |

### 功能应该放在哪一层？

| 功能 | 放哪里 | 原因 |
|------|--------|------|
| 打开/关闭串口 | AAR + UTS | 已有，不动 |
| 读/写字节 | AAR + UTS | 已有，不动 |
| HEX/Base64 转换 | UTS | 已有，不动 |
| 连接单例管理 | JS 服务层 | Vue 框架里更方便 |
| 帧解析(73...65) | JS 服务层 | 业务逻辑，纯 JS 就能做 |
| 事件订阅/分发 | JS 服务层 | 多页面共享数据 |
| 轮询定时器 | JS 服务层 | 集中管理 |
| UI 显示 | Vue 页面 | 展示层 |

### 关键结论

**不需要改 AAR，不需要改 UTS 插件！**

只需要在 JS 层加一个服务模块（serialService.js），它调用现有的 UTS 插件，然后提供更高级的封装：

```javascript
// serialService.js 核心思路

import { 
  openSerial,   // <-- 调用现有的 UTS 插件
  writeSerial, 
  readSerial, 
  closeSerial 
} from '@/uni_modules/wzl-serialbridge'

// 然后封装成更高级的 API
export function connect(path) {
  openSerial({ path, ... })  // 调用 UTS
}

export function send(data) {
  writeSerial({ ... })       // 调用 UTS
}

export function on(event, callback) {
  // 事件订阅机制
}
```

### 加了 JS 服务层之后的调用关系

```
serial-test.vue   ──调用──>  serialService.js  ──调用──>  UTS 插件
free-training.vue ─────────┘
其他页面 ─────────────────┘
```

好处：
- 串口连接全局唯一（单例）
- 轮询、帧解析只写一次
- 页面只需要订阅事件
- 页面切换不影响串口连接
- 关系：UTS 通过 import 语句导入 AAR 中的类，然后封装成 uni-app 风格的 API 供 JS 调用

### 数据流与事件设计

serialService.js 内部的数据流处理：

```
轮询 readSerial()
       |
       v
┌─────────────────────────────────────────────────────────┐
│  serialService.js                                       │
│                                                         │
│  每次读到数据 ──┬──> emit('raw', 原始数据)    ← 会拆包！ │
│                │                                        │
│                └──> rxBuffer 累积                       │
│                         │                               │
│                         v                               │
│                   检查帧头+长度+帧尾                    │
│                         │                               │
│                    完整帧？                             │
│                    /      \                             │
│                  是        否                           │
│                  │          │                           │
│                  v          v                           │
│        emit('frame', 完整帧)  继续等待                  │
│            ↓                                            │
│       只有完整帧！                                      │
└─────────────────────────────────────────────────────────┘
```

### 事件类型对比

| 事件 | 用途 | 是否拆包 | 适用场景 |
|------|------|---------|---------|
| `'raw'` | 调试用 - 观察底层读取情况 | 会拆包 | 串口调试页面 |
| `'frame'` | 业务用 - 获取解析后的完整帧 | 不会 | 正式业务页面 |
| `'connect'` | 连接成功通知 | - | 状态监听 |
| `'disconnect'` | 断开连接通知 | - | 状态监听 |
| `'error'` | 错误事件 | - | 异常处理 |

### 页面订阅示例

**调试页面（需要看原始数据）：**

```javascript
// serial-test-v2.vue
serial.on('raw', (data) => {
  // 每次轮询读到的原始碎片，可能不完整
  console.log('原始数据:', data.data, data.bytes, '字节')
})

serial.on('frame', (data) => {
  // 完整解析后的帧，永远是 96 字节
  console.log('完整帧:', data.parsed)
})
```

**正式业务页面（只关心完整帧）：**

```javascript
// free-training.vue
serial.on('frame', (data) => {
  // 只订阅 frame 事件，完全不需要关心拆包问题
  const force = data.parsed.left.instantForce
  const mode = data.parsed.setForceModeText
  
  // 更新 UI
  currentForce.value = force
  currentMode.value = mode
})
```

### 为什么保留 'raw' 事件？

虽然 `'raw'` 会显示拆包现象，但它有重要的调试价值：

1. **观察底层通信** - 确认数据是否正常到达
2. **排查硬件问题** - 如果 raw 没数据，说明是底层问题
3. **分析拆包规律** - 了解每次轮询读取的字节数
4. **对比帧解析效果** - 验证 frame 事件是否正确拼接

在正式业务页面中，完全忽略 `'raw'` 事件即可。

## 实践案例：定时器泄漏问题

### 问题背景

2026年2月9日，在开发串口测试页面时遇到一个典型 Bug：

**现象**：从 serial-test.vue（v1）页面跳转到其他页面后，HBuilderX 控制台不断打印错误：

```
10:35:47.096 读取失败:, {"errMsg":"串口未打开，请先调用 openSerial..."} at serial-test.vue:349
10:35:47.111 读取失败:, {"errMsg":"串口未打开，请先调用 openSerial..."} at serial-test.vue:349
10:35:47.142 读取失败:, {"errMsg":"串口未打开，请先调用 openSerial..."} at serial-test.vue:349
... (每 50ms 一条，无限循环)
```

### 问题分析

v1 页面直接调用 UTS 插件，页面自己管理轮询定时器：

```javascript
// v1 的实现方式（有问题）
const startReading = () => {
  readTimer = setInterval(() => {  // 页面自己创建定时器
    readSerial({ portId, ... })
  }, 50)
}

onBeforeUnmount(() => {
  stopStressTest()
  closeSerial({ portId })
  // 忘记调用 stopReading()！定时器还在后台运行！
})
```

当页面销毁时，`setInterval` 创建的定时器**并不会自动停止**，它仍在后台运行，不断调用 `readSerial`，而此时串口已关闭 → 报错。

### 根本原因

**手动管理定时器容易遗忘**。每个页面都要：
1. 创建定时器
2. 在 `onBeforeUnmount` 中清理定时器
3. 还要处理各种异常情况...

一旦漏掉任何一个地方，就会出 Bug。

### v1 vs v2 架构对比

| 对比项 | v1（直接调 UTS） | v2（通过 JS 服务层） |
|--------|-----------------|---------------------|
| **定时器管理** | 每个页面自己管理 | serialService 统一管理 |
| **页面销毁时** | 必须手动 stopReading() | 只需 off() 取消订阅 |
| **遗忘风险** | 高，容易忘记清理 | 低，服务层内部处理 |
| **定时器生命周期** | 跟随页面 | 跟随应用 |
| **代码复杂度** | 每页都写一遍 | 写一次，到处复用 |

### v1 的修复

```javascript
onBeforeUnmount(() => {
  stopReading()       // 必须加上这行！
  stopStressTest()
  closeSerial({ portId })
})
```

### v2 的设计

v2 使用 serialService.js 架构，页面不需要管理定时器：

```javascript
// v2 的实现方式（推荐）
import * as serial from '@/utils/serialService'

onMounted(() => {
  serial.on('frame', handleFrame)  // 订阅事件
})

onBeforeUnmount(() => {
  serial.off('frame', handleFrame) // 取消订阅即可
  // 不需要管定时器，serialService 内部处理
})
```

### 结论

这个 Bug 恰恰证明了 **JS 服务层架构的必要性**：

1. **职责分离**：定时器逻辑封装在服务层，页面专注 UI
2. **单一管理点**：定时器只在 serialService.js 里创建/销毁
3. **降低心智负担**：页面开发者不需要记住"离开页面要清理定时器"
4. **复用性**：多个页面共享同一个服务，不用重复写轮询代码

**v1 是临时方案，v2 才是正道！**

## ES Module 单例机制

### 为什么多个页面共享同一个 serialService？

在 uni-app（以及所有现代 JS 框架）中，ES Module 规范保证每个模块**只执行一次**，后续 `import` 都返回同一个模块对象的引用，而非拷贝。

```javascript
// serialService.js
let portId = 0           // 模块级变量
let isConnected = false  // 只存在一份

export function connect() { ... }
export function send() { ... }
```

```javascript
// 页面A
import serialService from '@/utils/serialService'
serialService.connect()  // portId 变成 1

// 页面B
import serialService from '@/utils/serialService'
console.log(serialService.getStatus().portId)  // 输出 1，不是 0
```

### 内存模型

```
+--------------------------------------------------+
|                    内存                           |
|                                                  |
|   serialService.js (单例)                         |
|   +------------------------------+               |
|   | portId = 1                   |               |
|   | isConnected = true           |               |
|   | listeners = { frame: [...] } |               |
|   | rxBuffer = "73..."           |               |
|   +------------------------------+               |
|         ^           ^           ^                |
|         |           |           |                |
|    +----+---+  +----+---+  +----+---+            |
|    | 页面 A |  | 页面 B |  | 页面 C |            |
|    | import |  | import |  | import |            |
|    +--------+  +--------+  +--------+            |
|    (引用)      (引用)       (引用)               |
+--------------------------------------------------+
```

### 单例设计要点

| 设计点 | 实现方式 |
|--------|----------|
| 单例连接 | 模块级 `portId`, `isConnected` |
| 事件广播 | `listeners` 数组，多页面订阅 |
| 状态共享 | `getStatus()` 返回当前状态 |
| 生命周期 | `cleanup()` 在 App 退出时调用 |

### 页面使用示例

```javascript
// freeTraining.vue
import serialService from '@/utils/serialService'

onMounted(() => {
  // 订阅帧事件
  serialService.on('frame', handleFrame)
  
  // 如果未连接，才连接
  if (!serialService.getStatus().isConnected) {
    serialService.connect()
  }
})

onUnmounted(() => {
  // 取消订阅，但不断开连接（其他页面可能还在用）
  serialService.off('frame', handleFrame)
})
```

**结论**：uni-app 中 `import` 模块天然就是单例，serialService.js 的设计充分利用了这一特性，无需额外的单例模式代码。

## serialService.js 协议解析实现

> 更新日期：2026年2月9日
> 
> 根据 Android 端 Java 解析代码，在 JS 服务层实现了完整的协议帧解析。

### 协议帧类型

本系统支持两种帧格式：

| 帧类型 | 包类型字节 | 方向 | 长度 | 用途 |
|--------|-----------|------|------|------|
| **A9 帧** | `0xA9` | 设备 → App | 96 字节 | 上报设备状态、传感器数据 |
| **D180 帧** | `0xB4` | App → 设备 | 9 字节 | 下发控制指令 |

### 帧结构

**通用帧格式：**
```
[帧头 0x73] [包类型] [序号] [数据区...] [CRC] [帧尾 0x65]
```

**A9 上行帧（96 字节）：**
```
偏移  长度  字段            说明
----  ----  --------------  ------------------------------------
[0]   1     帧头            固定 0x73
[1]   1     包类型          固定 0xA9
[2]   1     序号            0-255 循环

[3-4] 2     左手距离        mm (有符号 int16)
[5-6] 2     左手速度        mm/s
```

### 力量模式定义

| 值 | 模式名称 |
|----|---------|
| 0 | 关闭 |
| 1 | 恒力 |
| 2 | 离心等张 |
| 3 | 向心等张 |
| 4 | 弹力绳 |
| 5 | 流体阻力 |
| 6 | 平衡 |
| 7 | 绳索B1 |

### CRC 校验算法

```javascript
// 异或校验：从 [1] 开始到 CRC 前一位
function calculateCrc(bytes, crcOffset) {
  let crc = 0
  for (let i = 1; i < crcOffset; i++) {
    crc ^= bytes[i]
  }
  return crc
}
```

### 动态帧长度检测

serialService.js 实现了动态长度检测：

```javascript
// 读取包类型 [1]
const packType = parseInt(rxBuffer.substring(2, 4), 16)

// 根据包类型确定帧长度
if (packType === 0xA9) {
  frameHexLength = 192  // 96 * 2
  crcOffset = 94
} else if (packType === 0xB4) {
  frameHexLength = 18   // 9 * 2
  crcOffset = 7
}
```

```javascript
  sportCount: 50,
  
  temperature: {
    m0Mos: 45,          // 摄氏度
    m1Mos: 46,
    m0Motor: 50,
    m1Motor: 51
  },
  
  sleepState: 0,        // 0=未休眠
  
  crc: {
    field: 0x5A,        // 帧中的 CRC 值
    calc: 0x5A,         // 计算的 CRC 值
    valid: true
  },
  
  tail: 0x65
}
```

### D180 帧构建器

serialService.js 提供了便捷的帧构建方法：

```javascript
import { buildD180Frame, sendD180 } from '@/utils/serialService'

// 方式1：构建帧（返回 HEX 字符串）
const frameHex = buildD180Frame({
  force: 10,        // 力量 10kg
  forceMode: 1,     // 恒力模式
  lock: 0
})
// 输出: "73B4000A01007C65"

// 方式2：直接发送（自动构建并发送）
await sendD180({
  force: 10,
  forceMode: 1
})
```

### 事件数据结构

页面订阅 `'frame'` 事件时收到的数据：

```javascript
serial.on('frame', (data) => {
  console.log(data)
  // {
  //   raw: "73A9002AF3...65",  // 原始 HEX 字符串
  //   parsed: { ... },         // 解析后的对象（见上方）
  //   bytes: 96,               // 帧长度
  //   crcValid: true,          // CRC 校验结果
  //   ts: 1707451234567        // 时间戳
  // }
})
```

### 字节序说明

- 多字节数据采用 **大端序（Big-Endian）**
- 例如：距离 `0x00 0x96` 表示 `0x0096 = 150`

## API 文档

### 1. listDevices - 列举可用设备

```typescript
listDevices({
  prefixes: ["/dev/ttyS", "/dev/ttyUSB"], // 可选
  success: (res) => {
    console.log("可用设备:", res.devices)
    // 输出: ["/dev/ttyS0", "/dev/ttyS1", "/dev/ttyUSB0"]
  },
  fail: (err) => {
    console.error("失败:", err.errMsg)
  }
})
```

### 2. openSerial - 打开串口

```typescript
const options: OpenOptions = {
  path: "/dev/ttyS0",
  config: {
    baudRate: 115200,  // 波特率
    dataBits: 8,       // 数据位: 5|6|7|8
    stopBits: 1,       // 停止位: 1|2
    parity: 'none'     // 校验位: 'none'|'odd'|'even'
  },
  success: (res) => {
    const portId = res.portId
    console.log("打开成功,句柄ID:", portId)
    // 保存 portId 用于后续操作
  },
  fail: (err) => {
    console.error("打开失败:", err.errMsg)
  }
}

openSerial(options)
```

### 3. writeSerial - 写入数据

```typescript
// 写入 HEX 数据
const writeOptions: WriteOptions = {
  portId: 1,                    // openSerial 返回的句柄ID
  data: "01 03 00 00 00 01",   // HEX 字符串(自动去空格)
  format: 'hex',                // 'hex' | 'base64'
  timeout: 1000,                // 超时时间(毫秒)
  success: (res) => {
    console.log("写入成功:", res.bytesWritten, "字节")
  },
  fail: (err) => {
    console.error("写入失败:", err.errMsg)
  }
}

writeSerial(writeOptions)

// 或写入 Base64 数据
writeSerial({
  portId: 1,
  data: "SGVsbG8gV29ybGQ=",  // Base64 编码的数据
  format: 'base64',
  timeout: 1000,
  success: (res) => console.log("写入成功")
})
```

### 4. readSerial - 读取数据

```typescript
const readOptions: ReadOptions = {
  portId: 1,           // 句柄ID
  length: 1024,        // 最大读取字节数
  format: 'hex',       // 返回格式: 'hex' | 'base64'
  timeout: 1000,       // 超时时间
  success: (res) => {
    console.log("读取到数据:", res.data)
    console.log("字节数:", res.bytesRead)
    // HEX 格式输出: "010300000001"
    // Base64 格式输出: "AQMAAAAB"
  },
  fail: (err) => {
    console.error("读取失败:", err.errMsg)
  }
}

readSerial(readOptions)
```

### 5. closeSerial - 关闭串口

```typescript
closeSerial({
  portId: 1,
  success: (res) => {
    console.log("关闭成功")
  },
  fail: (err) => {
    console.error("关闭失败:", err.errMsg)
  }

```

## 完整使用示例

```vue
<template>
  <view class="container">
    <button @click="handleListDevices">列举设备</button>
    <button @click="handleOpen">打开串口</button>
    <button @click="handleWrite">写入数据</button>
    <button @click="handleRead">读取数据</button>
    <button @click="handleClose">关闭串口</button>
    <text>{{ statusMsg }}</text>
  </view>
</template>

<script setup lang="ts">
import { ref } from 'vue'
import { 
  openSerial, 
  writeSerial, 
  readSerial, 
  closeSerial,
  listDevices
} from '@/uni_modules/wzl-serialbridge'

const statusMsg = ref<string>('')
let currentPortId = 0

// 列举设备
const handleListDevices = () => {
  listDevices({
    success: (res) => {
      statusMsg.value = `找到设备: ${res.devices.join(', ')}`
    }
  })
}

// 打开串口
const handleOpen = () => {
  openSerial({
    path: "/dev/ttyS0",
    config: {
      baudRate: 115200,
      dataBits: 8,
      stopBits: 1,
      parity: 'none'
    },
    success: (res) => {
      currentPortId = res.portId
      statusMsg.value = `串口已打开,ID: ${currentPortId}`
    },
    fail: (err) => {
      statusMsg.value = `打开失败: ${err.errMsg}`
    }
  })
}

// 写入数据
const handleWrite = () => {
  if (currentPortId === 0) {
    statusMsg.value = "请先打开串口"
    return
  }
  
  writeSerial({
    portId: currentPortId,
    data: "01 03 00 00 00 01 84 0A",
    format: 'hex',
    success: (res) => {
      statusMsg.value = `写入成功: ${res.bytesWritten} 字节`
    },
    fail: (err) => {
      statusMsg.value = `写入失败: ${err.errMsg}`
    }
  })
}

// 读取数据
const handleRead = () => {
  if (currentPortId === 0) {
    statusMsg.value = "请先打开串口"
    return
  }
  
  readSerial({
    portId: currentPortId,
    length: 128,
    format: 'hex',
    timeout: 2000,
    success: (res) => {
      statusMsg.value = `读取到: ${res.data} (${res.bytesRead}字节)`
    },
    fail: (err) => {
      statusMsg.value = `读取失败: ${err.errMsg}`
    }
  })
}

// 关闭串口
const handleClose = () => {
  if (currentPortId === 0) {
    statusMsg.value = "串口未打开"
    return
  }
  
  closeSerial({
    portId: currentPortId,
    success: () => {
      statusMsg.value = "串口已关闭"
      currentPortId = 0
    },
    fail: (err) => {
      statusMsg.value = `关闭失败: ${err.errMsg}`
    }
  })
}
</script>
```

## 注意事项

1. **权限配置**: 需要在 `manifest.json` 中配置存储权限:
   ```json
   "android": {
     "permissions": [
       "<uses-permission android:name=\"android.permission.READ_EXTERNAL_STORAGE\"/>",
       "<uses-permission android:name=\"android.permission.WRITE_EXTERNAL_STORAGE\"/>"
     ]
   }
   ```

2. **自定义基座**: 由于使用了 AAR 库,需要制作自定义基座才能运行。

3. **设备路径**: Android 常见串口设备路径:
   - `/dev/ttyS0`, `/dev/ttyS1` - 系统串口
   - `/dev/ttyUSB0` - USB 转串口
   - `/dev/ttyAMA0` - 树莓派等设备

4. **数据格式**: 
   - HEX 格式支持空格分隔,如 `"01 03 00 00"`
   - Base64 格式需要标准 Base64 编码
